# 迭代开发规范

## 何时使用

| 场景 | 使用部分 |
|------|----------|
| 创建新文档或编写新内容 | 第二部分：文档书写规范 |
| 设计文档结构与层次 | 第一部分：技能结构与设计原则 |
| 让文档抵抗 AI 理性化跳过 | 第三部分：Bulletproofing 与反模式 |
| 验证文档有效性 | 第四部分：TDD 验证流程 |
| 修改现有文档或代码 | 第五部分：文档修改规范 |

---

## 核心原则

1. **Description = 触发条件**，绝不总结工作流程
2. **Token 效率优先**，按需加载，引用不重复
3. **验证驱动**，No document without failing test first

---

# 第一部分：技能结构与设计原则

## 1. Progressive Disclosure（三级加载）

技能文档使用三级加载系统，按需消耗 context：

| 级别 | 内容 | 大小限制 | 加载时机 |
|------|------|----------|----------|
| 1. 元数据 | name + description | ~100 字 | 始终在 context |
| 2. SKILL.md | 核心指令和工作流 | < 5000 字 | 技能触发时 |
| 3. 按需资源 | references/、scripts/、assets/ | 无限制 | Claude 判断需要时 |

**设计要点**：
- 元数据决定能否被发现 → description 必须精准
- SKILL.md 决定能否被理解 → 核心指令，不含细节
- references/ 提供详细信息 → 仅在需要时加载
- scripts/ 可直接执行 → 无需加载到 context

---

## 2. 文件组织

```
skill-name/
├── SKILL.md              # 必需：核心指令
├── scripts/              # 可执行脚本
├── references/           # 参考文档（按需加载）
└── assets/               # 输出资源（模板、图标等）
```

### 各目录用途

| 目录 | 加载方式 | 用途 |
|------|----------|------|
| `scripts/` | 执行或读取 | 需要确定性可靠性的重复代码 |
| `references/` | 按需读取 | API 文档、Schema、详细指南 |
| `assets/` | 输出时使用 | 模板、图片、字体 |

### 放置原则

| 内容类型 | 放置位置 |
|----------|----------|
| 核心工作流、原则 | SKILL.md 内联 |
| 重型参考（>100 行） | references/ |
| 可复用代码/工具 | scripts/ 或单独文件 |
| 输出用的模板/资源 | assets/ |

---

## 3. Token 效率设计

### 目标字数

| 文档类型 | 目标 | 原因 |
|----------|------|------|
| SKILL.md | < 500 字 | 高频加载 |
| eval-*.md | < 1000 字 | 阶段触发时加载 |
| references/*.md | 按需精简 | 按需加载但仍需精简 |

### 压缩技巧

| 技巧 | 说明 |
|------|------|
| 引用 --help | 脚本参数详情不重复记录，运行 --help 查看 |
| 交叉引用 | 重复内容引用其他文档，不复制 |
| 压缩示例 | 用最小示例说明模式，一个优秀示例足够 |
| 扁平化 | 三层嵌套 → 两层 |
| 表格化 | 冗长描述 → 表格 |
| 删除冗余 | 删除重复示例、声明性内容 |

### 交叉引用格式

```markdown
# ✅ 正确：明确是否必需
**REQUIRED BACKGROUND:** 参考 skill-creator/SKILL.md
**参考：** references/服务接口说明.md

# ❌ 错误：模糊
See skills/skill-creator/SKILL.md
@skills/skill-creator/SKILL.md  # 强制加载，浪费 context
```

---

## 4. 命名规范

### Skill Name

- 使用字母、数字、连字符
- 不使用括号、特殊字符
- 动名词形式适合流程：`creating-skills`、`testing-skills`
- 以动作或核心洞察命名：`condition-based-waiting` > `async-test-helpers`

### Description（CSO 优化）

**目的**：Claude 根据 description 决定是否加载技能。

**格式**：以 "Use when" 开头，只写触发条件。

**关键**：**绝不总结工作流程**。Description 总结流程会导致 Claude 跳过正文。

```yaml
# ❌ 错误：总结了流程 - Claude 会跳过正文
description: 执行计划时使用 - 为每个任务分发子代理，任务间进行代码审查

# ✅ 正确：只有触发条件
description: Use when executing implementation plans with independent tasks

# ❌ 错误：太抽象
description: 用于异步测试

# ✅ 正确：具体症状
description: Use when tests have race conditions, timing dependencies, or pass/fail inconsistently
```

# 第二部分：文档书写规范

## 1. YAML Frontmatter

所有阶段文档必须包含：

```yaml
---
name: eval-{stage}
description: Use when {触发条件}
---
```

---

## 2. 阶段文档结构

```markdown
---
name: eval-{stage}
description: Use when {触发条件}
---

# 阶段标题

## 目标
完成{目标清单}后，进入{下一阶段}。

核心原则：**{一句话}**。

## 何时使用
- {触发场景}

---

## 阶段完成标志

**验证顺序**（按序执行，任一失败则执行对应任务）：

1. ✅ {验证条件1} → 否则执行任务1
2. ✅ {验证条件2} → 否则执行任务2
...

全部通过后，进入{下一阶段}。

---

## 前置验证
- 必需文件列表

---

## 快速参考
| 任务 | 输出 | 检查点 |

---

## 任务列表

### 任务1：名称
**目标**：动作A → 动作B → 动作C。
**输出**：文件路径

---

## 常见错误
| 错误 | 原因 | 解决方案 |

---

## 变量速查
| 变量 | 说明 |
```

### 章节要求

| 章节 | 必需 | 说明 |
|------|------|------|
| YAML frontmatter | ✅ | name + description |
| 目标 | ✅ | 目标清单 + 核心原则 |
| 何时使用 | ✅ | 触发场景列表 |
| 阶段完成标志 | ✅ | 验证顺序，状态驱动 |
| 前置验证 | 按需 | 进入条件检查 |
| 快速参考 | ✅ | 任务概览表 |
| 任务列表 | ✅ | 按步骤执行 |
| 常见错误 | ✅ | 错误-原因-方案表 |
| 变量速查 | 按需 | **放在末尾** |

---

## 3. 语言风格

使用**命令式/不定式**（verb-first），不使用第二人称：

```
❌ 你应该检查文件是否存在
✅ 检查文件是否存在

❌ 如果用户需要调整，可以修改权重
✅ 用户调整时，修改权重
```

---

## 4. 任务目标描述

采用**过程导向**格式：

```markdown
❌ 产物导向：生成配置文件，询问用户调整。
✅ 过程导向：生成配置 → 展示摘要 → 等待确认。
```

### 多输出任务

```markdown
**输出**：
- `{work-dir}/.eval/{session-id}/eval-judge.json`（评委配置）
- 更新 `{work-dir}/.eval/{session-id}/eval-dimension.json` 的 `judge_id` 字段
```

### 用户确认约束

涉及用户确认的步骤必须标注：

```markdown
**步骤3**：确认配置
展示配置摘要并等待用户确认。

**注意**：此步骤必须等待用户确认，不可跳过。
```

---

## 5. 步骤判断逻辑

### 优先级

对话上下文 → 文件系统扫描

### 格式选择

| 分支数 | 格式 |
|--------|------|
| ≥3 | 表格（结构化） |
| 2 | 列表（紧凑） |

### 编号

使用"判断1、判断2"扁平结构，避免嵌套子判断。

```markdown
**判断1**：分析历史对话意图

| 用户意图 | 判断依据 | 后续动作 |
|----------|----------|----------|
| 新建评测任务 | 对话中提及"新建" | → 步骤3 |
| 无法判断 | 无历史对话 | → 判断2 |

**判断2**：扫描历史会话目录

- **存在** → 步骤2
- **不存在** → 步骤3
```

---

## 6. 流程图使用

**仅用于**：
- 非显而易见的决策点
- 可能过早停止的流程循环
- "何时使用 A vs B"的决策

**不用于**：
- 参考材料 → 用表格、列表
- 代码示例 → 用 Markdown 代码块
- 线性指令 → 用编号列表

---

## 7. 代码示例

**一个优秀示例胜过多个平庸示例**

| 要求 | 说明 |
|------|------|
| 完整可运行 | 示例可以直接执行 |
| 注释说明原因 | 解释 WHY，不只是 WHAT |
| 来源于真实场景 | 不是臆造的模板 |
| 清晰展示模式 | 一眼能看出核心模式 |

**禁止**：
- 用 5+ 种语言实现同一示例
- 创建填空式模板
- 编造的示例

# 第三部分：Bulletproofing 与反模式

## 1. 关闭漏洞

文档中声明规则后，必须显式禁止绕过方式：

```markdown
❌ 不够强：
写代码前先写测试？删除代码，重新开始。

✅ 关闭漏洞：
写代码前先写测试？删除代码，重新开始。

**无例外**：
- 不保留作为"参考"
- 不"一边写测试一边适配"
- 不查看它
- 删除就是删除
```

### 精神 vs 字面

在文档开头添加基础原则，防止"精神 vs 字面"论辩：

```markdown
**违反规则的字面意思就是违反规则的精神。**
```

---

## 2. 理性化表格

从测试中捕获 AI 的理性化借口，记录在表格中：

| 借口 | 现实 |
|------|------|
| "太简单不需要测试" | 简单代码也会出错，测试只需 30 秒 |
| "我会之后再测试" | 之后测试通过不能证明什么 |
| "之后再测试效果一样" | 先写测试 = "应该做什么"，后写测试 = "这做了什么" |
| "这是关于精神不是仪式" | 仪式确保精神被执行 |
| "这个情况不同因为..." | 每个违规都声称自己特殊 |

---

## 3. Red Flags

列出理性化的危险信号，让 AI 自检：

```markdown
## Red Flags - 停止并重新开始

- 代码写在测试之前
- "我已经手动测试过了"
- "之后测试效果一样"
- "这是精神问题不是仪式问题"
- "这个情况不同因为..."

**以上都意味着：删除代码，用 TDD 重新开始。**
```

---

## 4. 反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| 叙事示例 | "在会话 2025-10-03 中..." 太具体 | 提炼通用模式 |
| 多语言稀释 | example-js.js, example-py.py 质量平庸 | 一个优秀示例足够 |
| 流程图中写代码 | 无法复制粘贴 | 用 Markdown 代码块 |
| 通用标签 | helper1, step3 无语义 | 使用有意义的名称 |
| 描述总结流程 | AI 可能跳过文档正文 | 描述只写触发条件 |
| 冗长对话示例 | 占用大量 Token | 用最小示例说明模式 |
| 变量说明重复 | 占用额外 Token | 变量速查放在末尾 |

# 第四部分：TDD 验证流程

## 核心原则

**No document without failing test first.**

创建或修改文档前，必须先运行测试观察失败。

---

## TDD 映射

| TDD 概念 | 文档创建 |
|----------|----------|
| 测试用例 | 压力场景 + subagent |
| 生产代码 | 文档（SKILL.md 等） |
| RED（测试失败） | 无文档时 AI 违反规则（baseline） |
| GREEN（测试通过） | 有文档时 AI 遵守规则 |
| REFACTOR | 关闭漏洞，保持合规 |

---

## RED：Baseline 测试

**无文档运行压力场景**，记录 AI 的自然行为：

1. 选择压力场景（见下文）
2. 用 subagent 运行场景，**不加文档**
3. 记录 AI 的选择和理性化借口（逐字记录）
4. 识别违规模式

### 压力类型

| 压力类型 | 触发行为 |
|----------|----------|
| 时间压力 | 跳过步骤、合并任务 |
| 沉没成本 | "已经写了一半，继续用这个方案" |
| 权威压力 | "用户要求这样" |
| 疲劳 | 遗漏检查、简化流程 |
| 信息过载 | 遗漏关键检查 |
| 模糊输入 | 错误判断意图 |

### 压力场景示例

| 场景 | 验证目标 |
|------|----------|
| 时间压力 + 简单任务 | 是否跳过用户确认？ |
| 沉没成本 + 部分完成 | 是否删除重写？ |
| 模糊输入 + 无历史 | 是否正确判断？ |

---

## GREEN：写文档解决 Baseline 问题

**针对 baseline 中识别的具体问题编写文档**：

1. 只解决那些具体问题，不加假设性内容
2. 包含理性化表格中的借口对策
3. 添加 Red Flags 列表

**验证**：同一场景 + 文档，AI 现在应合规。

---

## REFACTOR：关闭漏洞

**AI 找到新的理性化借口？**：

1. 添加到理性化表格
2. 在文档中显式禁止
3. 重新测试直到防弹

---

## 测试不同类型文档

| 文档类型 | 测试方法 |
|----------|----------|
| 纪律强制型（规则/要求） | 学术问题 + 压力场景 + 组合压力 |
| 技术型（how-to） | 应用场景 + 变体场景 + 缺失信息 |
| 模式型（心智模型） | 识别场景 + 应用场景 + 反例 |
| 参考型（API 文档） | 检索场景 + 应用场景 + 覆盖测试 |

---

## 常见理性化借口

| 借口 | 现实 |
|------|------|
| "文档显然很清晰" | 对你清晰 ≠ 对其他 AI 清晰 |
| "只是参考文档" | 参考文档可能有漏洞 |
| "测试过度" | 未测试的文档有问题，15 分钟测试节省数小时 |
| "有问题再测" | 问题 = AI 无法使用文档，部署前测试 |
| "太繁琐" | 测试比调试生产中的问题更省时 |
| "我很自信" | 过度自信保证有问题 |
| "学术审查够了" | 阅读 ≠ 使用，测试应用场景 |
| "没时间测试" | 部署未测试的文档浪费更多时间 |

**所有这些意味着：部署前测试，无例外。**

# 第五部分：文档修改规范

## 1. 问题定位

| 问题阶段 | 目标文件 |
|----------|----------|
| 初始化 | `eval-init.md` |
| 构建阶段 | `eval-build.md` |
| 执行阶段 | `eval-execute.md` |
| 全局/调度 | `SKILL.md` |
| API/接口 | `references/服务接口说明.md` |
| 文件格式 | `references/中间产物说明.md` |
| 脚本 | `references/脚本定义.md` 或 `scripts/*.py` |

---

## 2. 修改原则

### 一致性

- **功能一致性**：修改前后功能一致（除非明确要求修改）
- **接口一致性**：脚本输入/输出保持兼容
- **风格一致性**：与现有文档风格一致

### 影响范围评估

| 修改类型 | 检查范围 |
|----------|----------|
| 步骤变更 | 上下游依赖 |
| 脚本参数 | 调用方文档 |
| 输出格式 | 后续步骤 |
| 模板结构 | 所有引用场景 |

### 向后兼容

- 新增字段：可选参数 + 默认值
- 废弃字段：标记 + 保留过渡期
- 结构变更：提供迁移方案

---

## 3. 修改检查清单

| 检查项 | 说明 |
|--------|------|
| 功能验证 | 修改后功能符合预期 |
| 依赖检查 | 是否影响引用 |
| 格式检查 | 符合书写规范 |
| 测试验证 | 相关用例通过 |

### 文档修改检查项

| 检查项 | 说明 |
|--------|------|
| YAML frontmatter | 符合 CSO 规范 |
| 结构一致性 | 符合阶段文档结构 |
| 目标格式 | 过程导向 |
| 确认约束 | 标注"不可跳过" |
| 跳转逻辑 | 无死循环或遗漏 |
| 引用完整性 | 文件/步骤存在 |
| 变量位置 | 速查在末尾 |

---

## 4. 脚本修改规范

### 原则

- 修改前后功能一致
- 保持接口兼容
- 变更必须说明影响

### 接口变更

```markdown
**变更说明**：
- 原接口：描述
- 新接口：描述
- 兼容性：是否向后兼容
- 迁移方案：不兼容时提供
```

---

## 5. 模板修改规范

### 原则

- 保持字段结构完整
- 新增字段设置默认值
- 废弃字段保留并标记

### 流程

1. 确认被引用位置
2. 评估影响范围
3. 执行修改
4. 更新相关文档
5. 通知使用方

---

## 6. 迭代记录

放置于 `iteration/` 目录，命名：`iteration-YYYY-MM-DD-标题.md`

```markdown
# 迭代标题

## 问题描述
现象和影响。

## 改进方案
修改内容和步骤。

## 影响范围
涉及的文件和功能。

## 验证结果
测试验证结果。
```

---

## 7. 常见问题

| 问题 | 现象 | 处理 |
|------|------|------|
| Token 失效 | API 401 | 引导重新授权 |
| 评测集格式不支持 | 解析报错 | 检查格式/编码/字段 |
| 任务执行失败 | 状态 Failed | 查询错误详情 |
| 维度配置校验失败 | dimension.py 报错 | 参考评测维度说明 |
