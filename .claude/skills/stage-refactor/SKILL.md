# 阶段重构专家

负责按照规范调整阶段文档（eval-init.md、eval-build.md、eval-set.md、eval-execute.md）。

## 核心原则

| 原则 | 说明 |
|------|------|
| 严格遵循规范 | 所有调整必须符合 `references/writing-guide.md` 定义的标准 |
| 流水线执行 | 按顺序完成各阶段，不得跳过 |
| 交互友好 | 每阶段完成后需用户确认再继续 |
| 允许逻辑调整 | 阶段文档支持任务顺序、步骤合并等逻辑调整（区别于流程文档） |

---

## 执行流程

### 第一阶段：分析现状

1. 读取目标阶段文档
2. 读取 `references/writing-guide.md` 规范
3. 对比分析，输出五维度差异报告：

**维度一：结构与格式**

| 章节 | 当前状态 | 规范要求 | 差异 |
|------|----------|----------|------|
| YAML frontmatter | ✓/✗ | name + description | ... |
| 目标章节 | ✓/✗ | 目标清单 + 核心原则 | ... |
| 前置验证 | ✓/✗ | 文件检查 + 失败处理 | ... |
| 阶段完成标志 | ✓/✗ | 验证顺序表格 | ... |
| 流程速查 | ✓/✗ | 编号/名称/位置/时机表格 | ... |
| 任务列表 | ✓/✗ | 任务→步骤层级 | ... |
| Red Flags | ✓/✗ | 违规行为表格 | ... |
| 常见错误 | ✓/✗ | 错误/原因/解决方案表格 | ... |
| 变量速查 | ✓/✗ | 引用 SKILL.md 或独立列出 | ... |

**维度二：交互友好性**

| 步骤 | 当前交互设计 | 问题 | 建议 |
|------|--------------|------|------|
| 步骤N | 是否有确认点？是否标注 ⚠️？ | 缺少确认/未说明目的 | 添加 ⚠️ 标注/补充说明 |

**维度三：内容精简性**

| 内容块 | 当前形式 | 问题 | 建议 |
|--------|----------|------|------|
| 字段列表/说明段落 | 列表/段落形式 | 可改为表格 | 改为表格形式 |

**维度四：内容逻辑合理性**

| 内容块 | 当前描述 | 问题 | 建议 |
|--------|----------|------|------|
| 任务顺序 | 任务N是否应前置/后置？ | 任务顺序不合理 | 调整任务顺序 |
| 步骤粒度 | 步骤是否应合并/拆分？ | 步骤过细/过粗 | 合并/拆分步骤 |
| 判断条件 | 条件覆盖是否完整？ | 遗漏分支/冗余判断 | 补充/精简判断 |
| 命令示例 | 参数是否正确？ | 参数过时/缺失 | 更新命令参数 |

> ⚠️ 此维度涉及业务逻辑调整，发现问题后应询问用户确认。

**维度五：阶段特有元素**

| 元素 | 当前状态 | 规范要求 | 差异 |
|------|----------|----------|------|
| 前置验证 | ✓/✗/缺失 | 检查前置文件，失败返回对应阶段 | ... |
| 阶段完成标志 | ✓/✗/缺失 | 验证顺序表格，状态驱动 | ... |
| 流程速查表 | ✓/✗/缺失 | 子流程引用汇总 | ... |
| 任务层级 | ✓/✗ | 三层结构（阶段→任务→步骤） | ... |

4. 询问用户："是否继续调整？"

---

### 第二阶段：确认调整范围

1. 展示拟调整内容清单（按五维度分类）
2. 询问用户调整方式：

| 选项 | 说明 |
|------|------|
| A. 全部调整 | 按规范完整调整所有差异项 |
| B. 选择性调整 | 用户指定调整哪些维度/项目 |
| C. 分步确认 | 每项调整前逐一确认 |

3. 特别确认：若涉及"内容逻辑合理性"维度调整，需单独确认：
   - 任务顺序调整：确认调整理由和新顺序
   - 步骤合并/拆分：确认合并/拆分方案
   - 判断条件修改：确认新增/删除的分支

4. 等待用户选择后进入下一阶段

---

### 第三阶段：执行调整

按用户确认的方式执行调整：

**全局要求**：
- 确保文档符合规范的全局目标
- 字数控制在 2000 字以内（阶段文档目标）

**结构调整**：
- 补充缺失章节（前置验证、阶段完成标志、流程速查）
- 调整章节顺序符合规范

**格式修正**：
- YAML frontmatter 格式
- 表格格式统一
- 步骤标注格式

**交互优化**：
- 为需要用户确认的步骤添加 ⚠️ 标注
- 说明交互目的

**内容精简**：
- 将列表/段落改为表格形式
- 去除冗余解释
- 控制示例数量

**逻辑调整**（用户确认后）：
- 调整任务顺序
- 合并/拆分步骤
- 更新判断条件
- 更新命令参数

每完成一项调整，输出：`✓ {调整项} 已完成`

## 判断依据

### 何时需要"前置验证"章节

| 场景 | 选择 |
|------|------|
| 非首阶段（eval-build/set/execute） | 必需前置验证 |
| 首阶段（eval-init） | 无前置阶段，可选 |

### 何时需要"流程速查"章节

| 场景 | 选择 |
|------|------|
| 阶段内有子流程调用（执行 processes/*.md） | 必需流程速查 |
| 阶段内无子流程调用 | 可省略 |

### 任务层级要求

- 阶段 → 任务 → 步骤（三层结构）
- 每个任务有明确目标
- 每个步骤有判断条件和动作

### Red Flags 数量要求

- 最少 3 条，最多 5 条
- 优先选择"不可跳过"和"不得提前询问"类错误

### 字数控制

- 阶段文档目标：< 2000 字
- 超出时优先精简示例和解释段落

---

## 与 process-refactor 的协同

| 技能 | 目标文档 | 允许逻辑调整 |
|------|----------|--------------|
| process-refactor | `processes/*.md` 流程文档 | ❌ 禁止 |
| stage-refactor | `eval-*.md` 阶段文档 | ✅ 允许（需用户确认） |

**协同场景**：阶段文档重构时，若发现子流程存在问题，可先完成阶段重构，再调用 process-refactor 处理子流程。

---

## 禁止行为

- ❌ 跳过用户确认阶段
- ❌ 未读取规范就执行调整
- ❌ 调整后未执行验证
- ❌ 未经用户确认修改任务顺序或步骤逻辑
- ❌ 删除必需章节（目标、完成标志、Red Flags）
