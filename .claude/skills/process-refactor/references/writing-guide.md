# 流程书写规范

本文档定义 processes 目录下流程文档的书写规范，确保结构统一、交互友好、内容精简。

---

## 全局要求

流程文档调整的核心目标：

| 目标 | 说明 |
|------|------|
| **结构清晰** | 章节顺序固定，必备章节不遗漏 |
| **步骤明确** | 目的 + 状态表，动作可执行 |
| **交互友好** | 用户知目的、选项有补充、确认标注 ⚠️ |
| **内容简洁** | 字数 < 1500，引用不重复 |
| **Token 效率** | 按需加载，引用外部文档 |
| **状态驱动** | 检测结果决定后续流程 |
| **一致性** | 与其他流程文档格式统一 |

---

## 一、宏观结构

### 1.1 必备章节顺序

| 序号 | 章节名称 | 必备性 | 说明 |
|------|----------|--------|------|
| 1 | YAML frontmatter | 必须 | name + description |
| 2 | 标题 + 流程描述 | 必须 | 一级标题 + 陈述式描述 |
| 3 | 目标 | 必须 | 流程目标 + 核心原则 |
| 4 | 触发条件 | 必须 | 简单一句描述 |
| 5 | 流程速查 | 可选 | 有子流程时必备 |
| 6 | 红线 | 必须 | 表格形式，置于流程速查后 |
| 7 | 步骤1-N | 必须 | 过程导向格式 |
| 8 | 产物 | 可选 | 有中间产物时必备 |
| 9 | 变量速查 | 必须 | 直接列出或引用 |

### 1.2 文档结构模板

```markdown
---
name: {流程名称}
description: {触发条件描述，不总结工作流程}
---

# {流程名称}

{陈述式流程描述，体现目的与效果}

## 目标

{流程目标}。

核心原则：**{核心原则}**。

---

## 触发条件

{触发条件描述，简单一句}。

---

## 流程速查（可选）

| 编号 | 名称 | 调用时机 |
|------|------|----------|
| 流程X.1 | {子流程名称} | {调用时机} |

---

## 红线

| 错误 | 原因 | 解决方案 |
|------|------|----------|
| {错误描述} | {错误原因} | {解决方案} |

---

## 步骤1：{步骤名称}

**目的**：{步骤目的}。

| 状态 | 动作 |
|------|------|
| {状态描述} | → {后续动作} |

{步骤详细内容}

---

## 步骤N：{步骤名称}

...

---

## 产物（可选）

| 文件 | 用途 |
|------|------|
| {文件名} | {用途说明} |

---

## 变量速查

| 变量 | 说明 |
|------|------|
| `{变量名}` | {变量说明} |

或：见 [SKILL.md 变量速查](../SKILL.md#变量速查)。
```

## 二、细节要求

### 2.1 YAML Frontmatter

| 字段 | 要求 |
|------|------|
| `name` | 流程英文名称，与文件名一致 |
| `description` | 触发条件描述，**不总结工作流程** |

**正确示例**：
```yaml
---
name: evalset-parse
description: Use when user provides evalset file and needs parsing to standard format
---
```

**错误示例**：
```yaml
description: 解析评测集、检测状态、调用子流程并返回标准化阶段的完整流程
```
> ❌ 错误：总结了工作流程，而非触发条件。

---

### 2.2 标题与流程描述

| 要求 | 说明 |
|------|------|
| 标题格式 | `# {流程名称}` |
| 流程描述位置 | 标题下方，目标章节上方 |
| 流程描述风格 | 陈述式，体现目的与效果 |
| 流程描述字数 | 一句话，不超过50字 |

---

### 2.3 目标章节

| 要求 | 说明 |
|------|------|
| 流程目标 | 使用箭头符号表示流程走向 |
| 核心原则 | 加粗标注 |

**示例**：
```markdown
## 目标

解析评测集 → 检测状态 → 调用子流程 → 返回标准化阶段。

核心原则：**步骤顺序执行，检测结果决定子流程**。
```

---

### 2.4 触发条件章节

| 要求 | 说明 |
|------|------|
| 格式 | 简单一句描述 |
| 内容 | 描述用户行为或系统状态，**不涉及具体阶段/任务/步骤编号** |
| 字数 | 不超过50字 |

**正确写法**：
```markdown
## 触发条件

用户指明用例级评测，且未匹配到该评测方式下的专家模板。
```

**错误写法**：
```markdown
## 触发条件

构建阶段任务2步骤3，用户选择定制用例级评测。
```
> ❌ 错误：涉及具体阶段、任务、步骤编号，这是"位置"而非"条件"。

---

### 2.5 流程速查章节

| 要求 | 说明 |
|------|------|
| 适用场景 | 有子流程调用时必备 |
| 表格格式 | 编号 + 名称 + 调用时机 |

---

### 2.6 红线章节

| 要求 | 说明 |
|------|------|
| 位置 | 流程速查章节后（无流程速查时置于触发条件后） |
| 格式 | 表格形式：错误 + 原因 + 解决方案 |
| 条数 | 3-5条核心错误 |

---

### 2.7 步骤章节

| 要求 | 说明 |
|------|------|
| 命名格式 | `## 步骤N：{步骤名称}` |
| 必备元素 | **目的**说明 + 状态表 |
| 状态表格式 | 状态 → 动作，动作列加 `→` 符号 |
| 确认标注 | 不可跳过的确认步骤标注 `⚠️` |

---

### 2.8 产物章节

| 要求 | 说明 |
|------|------|
| 必备性 | 可选，有中间产物时必备 |
| 格式 | 表格形式：文件 + 用途 |

---

### 2.9 变量速查章节

| 形式 | 适用场景 |
|------|------|
| 直接列出 | 流程专用变量 |
| 引用 SKILL.md | 通用变量（work-dir、session-id 等） |

---

### 2.10 文档引用格式

| 场景 | 格式 |
|------|------|
| 引用 .md 文档 | `` `文档名.md` `` |
| 引用 .json 文件 | `` `文件名.json` `` |
| 引用路径 | `` `路径/文件名` `` |

**正确写法**：`` 必须 `维度配置转化规则.md` 转化 ``

**错误写法**：`` 必须按维度配置转化规则.md 转化 ``

---

## 三、检查维度

### 3.1 结构与格式检查

| 检查项 | 说明 |
|--------|------|
| YAML frontmatter | name + description 符合规范 |
| 章节顺序 | 按必备章节顺序排列，无遗漏 |
| 流程描述 | 标题下方有一句陈述式描述 |
| 触发条件 | 简单一句，语义化条件 |
| 红线位置 | 流程速查后，表格形式，3-5条 |
| 步骤格式 | 有目的 + 状态表，动作列有 → |
| 字数控制 | < 1500字 |

---

### 3.2 交互友好性检查

| 检查项 | 问题信号 | 解决方案 |
|--------|----------|----------|
| 确认点标注 | 步骤涉及用户选择/调整内容但无 ⚠️ 标注 | 添加 ⚠️ 标注 + 说明交互目的 |
| 目的说明 | 执行动作前未说明"为什么" | 添加"**目的**"说明 |
| 选项清晰度 | 选项描述模糊，用户可能困惑 | 补充说明或提供示例 |
| 状态表完整性 | 只有文字描述，无状态 → 动作映射 | 改为状态表格式 |

---

### 3.3 内容精简性检查

| 检查项 | 问题信号 | 解决方案 |
|--------|----------|----------|
| 列表改表格 | 字段列表、属性列表超过3项 | 改为表格形式 |
| 段落改表格 | 描述性段落包含多个对比项 | 改为对比表格 |
| 内容重复 | 同一信息在不同步骤重复出现 | 使用引用代替复制 |
| 过度解释 | 规则本身已清晰，仍有额外解释 | 删除冗余解释 |
| 示例过多 | 每个细节要求超过1个示例 | 保留1个典型示例 |

---

### 3.4 内容逻辑合理性检查

| 检查项 | 问题信号 | 处理方式 |
|--------|----------|----------|
| 建议方案合理 | 默认值/推荐方案使用通用假设（如"均分"） | 标注问题，询问用户业务实际 |
| 示例准确 | 示例过于简化或偏离真实场景 | 标注问题，询问用户确认 |
| 业务约束体现 | 遗漏关键业务规则或限制条件 | 标注问题，询问用户补充 |
| 用户选择依据 | 用户做选择时缺少判断标准或信息 | 补充判断依据或提供默认方案 |

> ⚠️ **重要**：内容逻辑问题需结合业务场景判断，重构时发现问题应询问用户确认，不可凭通用直觉直接修改。

---

## 四、判断依据

### 4.1 变量速查形式选择

| 场景 | 选择 |
|------|------|
| 变量仅本流程使用 | 直接列出 |
| 变量为通用变量（work-dir、session-id 等） | 引用 SKILL.md |
| 变量混合存在 | 通用变量引用 + 专用变量列出 |

### 4.2 流程速查章节判断

**有子流程时必备**，子流程判断标准：
- 文档中明确调用其他流程文件（如"执行流程3.1"）
- 步骤中存在"调用 xxx 流程"的动作

### 4.3 红线条数要求

- 最少 3 条，最多 5 条
- 优先选择"必须等待用户确认"和"不得凭直觉判断"类错误

### 4.4 触发条件字数

- 一句话，不超过 50 字
- 不重复 description，但可细化场景

---

## 五、字数目标

| 文档类型 | 目标字数 | 原因 |
|----------|----------|------|
| 流程文档 | < 1500字 | 按需加载，保持精简 |
