# eval-set.md 任务拆分设计文档

## 问题描述

当前 `eval-set.md` 只有 **1个任务（任务1：处理评测集）** 包含 5个步骤，与其他阶段文档结构不一致：

| 阶段 | 任务数 | 每任务步骤数 |
|------|--------|--------------|
| eval-init | 3 | 2-4 步 |
| eval-build | 3 | 2-4 步 |
| eval-set | **1** | **5 步** |
| eval-execute | 3 | 1-3 步 |

**问题**：
1. 单任务过于庞大，包含多个用户交互点和分支逻辑
2. 与其他阶段结构不一致，影响整体协调性
3. 任务边界不清晰，难以追踪进度

---

## 改进方案

**推荐方案：按交互边界拆分（4个任务）**

| 任务 | 内容 | 用户交互点 | 输出文件 |
|------|------|------------|----------|
| 任务1：获取评测集 | 步骤1（判断+可能调用evalset-create） | 有（询问是否有评测集） | 可能调用 evalset-create |
| 任务2：解析与预处理 | 步骤2（解析+状态判断+流程7/8） | 有（partial分支询问） | evalset-status.json + 流程7/8输出 |
| 任务3：标准化与评测点 | 步骤3-4（标准化+评测点生成） | 无（纯脚本/条件执行） | evalset-standard.jsonl + keypoints.json |
| 任务4：上传评测集 | 步骤5（上传） | 无（纯脚本执行） | evalset-meta.json |

**理由**：
1. eval-set 的复杂性更高（多个分支判断），细粒度拆分更清晰
2. 与 eval-init 模式接近（eval-init 也有细粒度技术操作拆分）
3. 每个任务有明确的输出文件，便于状态追踪

#### 任务2：解析与预处理

**目标**：解析评测集 → 状态判断 → 执行对应预处理流程。

**步骤列表**：
1. 执行解析流程
2. 状态判断与分支处理

**步骤1：执行解析流程**

执行流程3（评测集解析），产出 `evalset-status.json`。

**步骤2：状态判断与分支处理**

读取 `evalset-status.json` 中 `answer.status`，按表格执行：

| 状态 | 含义 | 后续处理 |
|------|------|----------|
| `all_empty` | answer 全空 | → 执行流程7 → 任务3 |
| `all_filled` | answer 全非空 | → 执行流程8 → 任务3 |
| `partial` | answer 部分填充 | → 询问用户 → 执行对应流程 → 任务3 |

**partial 分支处理**：

询问用户："answer 字段部分填充，请确认评测集类型："
- 只有问题 → 执行流程7 → 任务3
- 问题+答案 → 执行流程8 → 任务3
