# Harness Design Skill 设计规范

## 概述

创建一个通用型 Harness 工程 Skill，让 Claude Code 在处理任何任务时（代码开发、Skill 开发、项目重构、内容生成等），都能遵循 Harness 工程思想，确保任务可控、可靠。

---

## 设计目标

| 目标 | 说明 |
|------|------|
| **通用性** | 适用于任何任务类型，不预置场景模板 |
| **动态分析** | 根据当前任务特征动态判断所需机制 |
| **方法论导向** | 教"如何设计机制"，而非"场景该怎么做" |
| **可控可靠** | 通过 Planner/Evaluator/Contract 等机制保障质量 |

## SKILL.md 内容规范

### YAML Frontmatter

```yaml
---
name: harness-design
description: Design quality assurance mechanisms for any task. Use when user invokes '/harness-design', asks for "harness engineering", "Planner", "Evaluator", "Sprint Contract", "Context Reset", or mentions "task reliability", "quality control", "make task controllable" for code development, skill development, refactoring, content generation, or presentation tasks.
---
```

### 核心流程

```
触发 → 5维度诊断 → 决策表定位 → 动态设计机制 → 输出选择
```

### 5维度诊断

| 维度 | 判断依据 | 等级划分 |
|------|----------|----------|
| **复杂度** | 改动文件数、模块数、依赖关系 | 简单 / 中等 / 复杂 |
| **主观性** | 是否有二元验证标准 vs 需人为判断 | 客观 / 半主观 / 高主观 |
| **验收清晰度** | 用户是否给出可验证的交付标准 | 清晰 / 模糊 / 缺失 |
| **任务时长** | 预计操作步骤数、文件处理量 | 短 / 中 / 长 |
| **迭代预期** | 用户是否要求多方案或暗示需打磨 | 单次 / 需打磨 / 持续 |

### 决策表

| 机制 | 触发条件 | 详细方法论 |
|------|----------|------------|
| **Planner** | 复杂度 ≥ 中等 | references/mechanisms/planner.md |
| **Evaluator** | 主观性 ≥ 半主观 或 任务重要 | references/mechanisms/evaluator.md |
| **Sprint Contract** | 验收清晰度 ≤ 模糊 | references/mechanisms/sprint-contract.md |
| **Context Reset** | 任务时长 ≥ 长 | references/mechanisms/context-reset.md |
| **G-E 循环** | 迭代预期 ≥ 需打磨 | 见 evaluator.md |

**重要任务强制规则**：生产环境、安全、资金相关 → 必须启用 Evaluator

### 动态设计机制

定位所需机制后，根据当前具体任务设计各机制参数：
- Planner：分解策略、子任务边界、验收标准
- Evaluator：评估维度、评分标准、校准方式
- Contract：协商重点、验收项、边界定义
- Context Reset：触发时机、Artifact 内容

### 输出选择

| 形式 | 产出 |
|------|------|
| 仅参考建议 | 设计建议文档 |
| 立即执行 | Claude 进入 Harness 循环执行 |
| 详细咨询 | 深入讨论某个机制 |

### 纠正模式

任务进行中触发时，先诊断问题：

| 问题表现 | 原因 | 处理 |
|----------|------|------|
| 输出质量虚高 | 自我评估偏差 → 添加/加强 Evaluator |
| 反复修正同一问题 | 理解偏差 → 添加 Contract 重新协商 |
| 提前结束/收尾倾向 | 上下文焦虑 → 检查是否需要 Context Reset |
| 任务执行混乱 | 缺少分解 → 添加 Planner 重新规划 |

## mechanisms/planner.md 内容规范

### 核心理念

复杂任务先分解再执行，避免一次性处理导致失控。

### 判断条件

| 判断依据 | 需要 Planner | 不需要 Planner |
|----------|--------------|----------------|
| 改动文件数 | > 3 文件 | ≤ 3 文件 |
| 模块数 | 涉及多模块 | 单模块内 |
| 依赖关系 | 有跨模块依赖 | 无或简单依赖 |

### 设计要点

1. **分解粒度**：每个子任务可独立验收、1-3 小时可完成、依赖关系显式标注
2. **分解边界**：功能边界、模块边界、时间边界、验收边界
3. **验收标准**：可验证、具体、可达成

### 输出结构

```markdown
## 任务分解

**总目标**：{当前任务的目标}

**子任务列表**：

| 序号 | 子任务 | 涉及范围 | 依赖 | 产出 | 验收标准 |

**关键决策点**：{需要先确认的设计决策}
**风险点**：{可能导致偏离的问题}
```

### 设计原则

| 原则 | 说明 |
|------|------|
| 高层不指定细节 | 只定义目标和验收，不指定实现方案 |
| 依赖显式标注 | 明确执行顺序 |
| 每个子任务独立验收 | 有产出可验证 |
| 边界清晰 | 包含/不包含明确 |

### Common Pitfalls

| Pitfall | 症状 | 原因 | 解决方案 |
|---------|------|------|----------|
| **分解粒度过细** | 子任务过多（> 15 个），执行效率低 | 过度拆分简单任务 | 合并相关子任务，保持 1-3 小时粒度 |
| **分解粒度过粗** | 子任务无法独立验收或耗时过长 | 拆分不足 | 按功能/模块边界进一步拆分 |
| **验收标准模糊** | "功能完成"等不可验证表述 | 未转化为具体行为 | 必须有可测试/可检查的验收项 |
| **依赖关系遗漏** | 执行时发现前置条件未满足 | 未显式标注依赖 | 检查所有子任务的前置条件 |
| **过度指定实现** | Planner 直接指定代码实现细节 | 越权决策 | Planner 只定义目标和验收，实现交给 Generator |

## mechanisms/evaluator.md 内容规范

### 核心理念

生成者与审查者分离，防止自我评估偏差导致质量虚高。

### 判断条件

| 条件 | 强度 |
|------|------|
| 主观性 = 高主观 | **必须** |
| 任务涉及生产/安全/资金 | **必须** |
| 用户明确表示重要 | **必须** |
| 用户反复修正同一问题 | **必须** |
| 主观性 = 半主观 | 推荐 |
| 任务时长 > 1 小时 | 推荐 |

### 设计三要素

1. **评估维度**：将模糊判断转化为可评分维度
   - 从"好不好" → 具体维度
   - 每个维度独立评分

2. **评分标准**：为每个维度定义评分等级
   - 每个等级有具体表现描述
   - 至少一个维度有硬阈值
   - 权重偏向 Claude 默认表现弱的维度

3. **校准机制**：防止评分漂移
   - Few-shot 示例
   - 检查清单
   - 硬阈值

### G-E 循环

```
Generator 执行 → Evaluator 评估 → 判断 → Generator 改进 → 循环
```

| 参数 | 设计依据 |
|------|----------|
| 最大轮次 | 根据迭代预期设定（默认 5） |
| 退出条件 | 所有维度 ≥ 硬阈值 |
| 策略转向 | 连续无进展时可转向新方案 |

### 输出结构

```markdown
## 评估报告

**维度评分**：
| 维度 | 分数 | 等级 | 具体问题 |

**硬阈值检查**：{通过/不通过}
**总评**：{通过/不通过}

**问题清单**（按优先级）：
| 序号 | 问题 | 改进建议 |
```

### Common Pitfalls

| Pitfall | 症状 | 原因 | 解决方案 |
|---------|------|------|----------|
| **评分过宽容** | 明显问题仍获高分 | Claude 倾向正面评价 | 添加 skeptical 角度的 prompt，强调批判性 |
| **维度过于抽象** | "整体效果"不可评分 | 未转化为具体特征 | 每个维度必须有可观察的具体指标 |
| **缺少硬阈值** | 所有维度都软评分，易妥协 | 未设置强制失败条件 | 至少一个维度有硬阈值，低于则强制失败 |
| **反馈不可执行** | "整体不错"无改进方向 | 未给出具体建议 | 每个问题必须配具体可执行的改进建议 |
| **评分漂移** | 相似产出评分不一致 | 缺少校准参照 | 添加 few-shot 示例或检查清单 |
| **过早批准** | 未发现明显 bug 就通过 | 测试 superficially | 强调 probing edge cases，添加深度检查要求 |

## mechanisms/sprint-contract.md 内容规范

### 核心理念

执行前先协商"完成定义"，防止理解偏差导致无效迭代。

### 判断条件

| 条件 | 说明 |
|------|------|
| 验收清晰度 = 模糊 | 用户需求如"好看"、"好用"等 |
| 验收清晰度 = 缺失 | 用户只说"帮我做个 X"无具体要求 |
| 任务重要 | 涉及生产/安全/资金等 |
| 用户反复修正 | 暗示此前理解有偏差 |

### 设计要点

1. **验收标准设计**：可验证
   - 转化模糊表述为具体验收项
   - 每个验收项有验证方式

2. **边界定义**：明确包含和不包含范围

3. **协商流程**：
   ```
   Generator 提议 → Evaluator/用户审核 → 判断 → 达成一致
   ```

### 输出结构

```markdown
## Sprint Contract

**目标**：{本次要完成的目标}

**构建内容**：
| 序号 | 内容 | 方案摘要 |

**验收标准**（必须全部满足）：
| 序号 | 验收项 | 验证方式 |

**不包含范围**：
- {明确排除的内容}

**风险点**：{可能导致偏离的问题}
```

### 设计原则

| 原则 | 说明 |
|------|------|
| 验收可验证 | 能测试/能检查 |
| 边界明确 | 包含/不包含清晰 |
| 保持精简 | 核心验收项 ≤ 5 |
| 协商后再执行 | 未达成不可执行 |

### Common Pitfalls

| Pitfall | 症状 | 原因 | 解决方案 |
|---------|------|------|----------|
| **验收标准模糊** | "功能完成"等不可验证表述 | 未转化为具体行为 | 必须有可测试/可检查的验收项和验证方式 |
| **边界不明确** | 执行中范围蔓延 | 未明确不包含什么 | 必须列出"不包含范围" |
| **Contract 过于复杂** | 验收项过多（> 7 个） | 过度细化 | 精简为核心验收项 ≤ 5 |
| **跳过协商直接执行** | Generator 未等用户确认就开始 | 忽略协商流程 | 强制：未达成一致不可执行 |
| **Contract 不更新** | 需求变更后仍按旧 Contract 执行 | 未及时重新协商 | 发现需求变化立即重新协商 |

## mechanisms/context-reset.md 内容规范

### 核心理念

清空上下文窗口，通过 Artifact 传递状态，防止上下文焦虑导致过早收尾。

### Compaction vs Context Reset

| 方式 | 操作 | 效果 |
|------|------|------|
| Compaction | 原地压缩历史为摘要 | 保持同一 agent，焦虑可能残留 |
| Context Reset | 清空 context，启动新 agent | 干净起点，需 Artifact 传递状态 |

### 判断条件

| 条件 | 说明 |
|------|------|
| 任务时长 = 长 | 预计 > 50 步操作或几小时 |
| 上下文填充度 > 70% | Claude 出现收尾倾向 |
| Generator 提前结束 | "差不多了，剩下的你自己处理" |
| 跨 session 执行 | 用户中断后恢复 |

### 设计要点

1. **触发时机**：填充度检测、收尾倾向、预估时长

2. **Artifact 内容**：
   - 任务背景（为什么做）
   - 当前状态（做到哪里）
   - 下一步指引（怎么做）
   - 关键决策（已做决策）
   - 注意事项（执行注意点）

3. **验证衔接点**：已完成产出存在、内容符合预期、无断裂遗漏

### 输出结构

```markdown
# Harness State Artifact

**生成时间**：{timestamp}
**任务目标**：{当前任务的目标}

## 执行进度

**当前阶段**：{阶段名}

**已完成**：| 序号 | 任务 | 产出 | 状态 |
**待完成**：| 序号 | 任务 | 依赖 | 预计产出 |

## 关键决策

| 决策点 | 决策内容 | 原因 |

## 下一步

**正在处理**：{当前任务摘要}
**下一步操作**：{具体指引}

## 注意事项

- {执行过程中的特殊注意点}
```

### 设计原则

| 原则 | 说明 |
|------|------|
| Artifact 自描述 | 新 Agent 无历史 context 也能理解 |
| 验证衔接点 | 新 Agent 验证已完成产出 |
| Harness 连续性 | Reset 后继续相同机制 |
| 决策已记录 | 关键决策记录在 Artifact |

### Common Pitfalls

| Pitfall | 症状 | 原因 | 解决方案 |
|---------|------|------|----------|
| **Artifact 信息不足** | 新 Agent 无法理解上下文 | 缺少背景/状态/指引 | Artifact 必须包含：任务背景、当前状态、下一步指引 |
| **未验证衔接** | 假设已完成产出存在 | 跳过验证步骤 | 新 Agent 启动后必须验证文件存在和内容 |
| **Harness 中断** | Reset 后跳过 Evaluator/Contract | 未在 Artifact 中要求 | 明确记录需要继续的 Harness 机制 |
| **决策未记录** | 新 Agent 重复做决策 | 关键决策未写入 Artifact | 所有设计决策必须记录 |
| **Reset 时机错误** | 过早或过晚 Reset | 未监控上下文状态 | 填充度 > 70% 或出现收尾倾向时触发 |
| **Artifact 过大** | Token 消耗过多 | 记录过多细节 | 保持精简，只记录关键信息 |

## examples/diagnosis-examples.md 内容规范

### 目的

展示如何对不同任务进行 5 维度诊断。是**诊断过程示例**，不是场景模板。

### 示例结构

每个示例包含：

```markdown
## 示例N：{任务类型}

**用户需求**：{具体需求描述}

**诊断过程**：

| 维度 | 分析 | 结论 |

**诊断结果**：{5维度结论}

**所需机制**：{机制列表}
```

### 关键洞察总结

| 维度组合 | 决定机制组合 |
|----------|--------------|
| 简单 + 客观 + 清晰 | 无 Harness |
| 高主观 | 必须 Evaluator |
| 模糊/缺失 | 必须 Contract |
| 复杂 | 必须 Planner |
| 长 | 必须 Context Reset |
| 需打磨 | 必须 G-E 循环 |

**核心原则**：根据当前任务的具体特征动态判断，不预设场景模板。
