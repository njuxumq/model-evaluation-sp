# 进度展示优化设计

## 概述

优化 model-evaluation skill 的进度展示机制，解决以下问题：
- 进度展示规则在各文档中重复定义
- 展示格式不统一，维护困难
- 缺少全局视角的进度概览

## 设计目标

1. **集中定义**：进度展示规则仅在 SKILL.md 定义一次
2. **分层展示**：SKILL → 阶段 → 任务 → 步骤，每层展示必要信息
3. **路径可视**：面包屑式路径 + 当前层级展开，清晰展示进度
4. **文档精简**：各阶段文档专注业务内容，不定义展示格式

---

## 进度展示规范

### 展示层级

| 时机 | 展示内容 |
|------|----------|
| 进入 SKILL | 全流程概览（阶段列表 + 每阶段任务列表） |
| 进入阶段 | 阶段路径 + 当前阶段任务列表 |
| 进入任务 | 任务路径 + 当前任务步骤列表 |
| 进入步骤 | 步骤路径 + 分支判断/步骤描述 |

### 状态符号

| 符号 | 状态 |
|------|------|
| `✓` | 已完成 |
| `●` | 当前 |
| `○` | 待执行 |
| `~` | 已跳过 |

### 分支判断格式

**展示判断条件**：

```
{判断条件}
  ├─ {选项A} → {后果A}
  ├─ {选项B} → {后果B}
  └─ {选项C} → {后果C}
```

**展示判断结果**：

```
📋 判断结果：{结果} → {后续动作}
```

## 阶段文档规范

### 文档结构

各阶段文档（eval-init.md、eval-build.md 等）遵循以下结构：

```markdown
---
name: eval-{stage}
description: 触发条件描述
---

# {阶段名称}

## 目标

{阶段目标描述}

## 任务列表

### 任务1：{任务名称}

**步骤列表**：
1. {步骤1名称}
2. {步骤2名称}
3. {步骤3名称}

#### 步骤1：{步骤名称}

**判断**：
- **条件A** → {动作A}
- **条件B** → {动作B}

{步骤操作说明}

#### 步骤2：{步骤名称}

...
```

### 删除的内容

从各阶段文档中删除以下内容：
- "进入任务时输出" 格式定义
- "进入步骤时输出" 格式定义
- 进度追踪方块（`┌─────┐` 格式）
- 路径记忆格式定义

### 保留的内容

各阶段文档保留以下内容：
- 任务列表
- 步骤列表
- 判断条件
- 操作说明
- 命令示例
- Red Flags

```markdown
### 任务1：环境及依赖检测

#### 步骤1：检查环境缓存文件

**步骤1判断**：

```

  ├─ 是 → 验证缓存有效性

```

### 任务1：环境及依赖检测

**步骤列表**：
1. 检查缓存
2. 验证缓存
3. 执行检测

检查 `{work-dir}/.eval/env.cfg` 是否存在。

- **文件不存在** → 进入步骤3
```

## 预期效果

1. **Token 效率提升**：减少重复的格式定义，降低文档总 Token 量
2. **维护性提升**：进度展示规则集中在一处，修改时只需更新 SKILL.md
3. **一致性提升**：所有阶段使用统一的展示格式，用户体验一致
4. **可读性提升**：面包屑路径 + 任务列表，用户可清晰了解当前进度
