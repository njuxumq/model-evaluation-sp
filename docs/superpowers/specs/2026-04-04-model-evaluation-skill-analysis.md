# 模型评测技能结构分析与改进方案

> 版本：v2.0 | 日期：2026-04-04

## 一、问题重新定义

### 1.1 核心问题

**问题本质**：文档职责边界模糊，导致内容堆砌和 Token 浪费

**表现形式**：
- SKILL.md 承担了"入口 + 规范 + 速查 + 目录说明"多重职责
- eval-set.md 同时定义"做什么"和"怎么做"
- 流程文档缺少强制性的门控机制

### 1.2 设计目标

| 目标 | 说明 | 度量标准 |
|------|------|----------|
| **职责分明** | 每个文档有单一明确的职责 | 一句话能描述文档用途 |
| **Token 高效** | 按需加载，不加载不需要的内容 | 入口 <500字，阶段 <2000字，流程 <1500字 |
| **结构清晰** | 层级分明，易于导航 | 最多 3 层嵌套 |
| **步骤明确** | 执行步骤无歧义 | 每步骤有明确的输入输出 |
| **交互友好** | 关键点有确认机制 | HARD-GATE 标注 |
| **内容简洁** | 无冗余重复 | 引用代替复制 |

## 二、文档层级架构设计

### 2.1 三层架构模型

```
┌─────────────────────────────────────────────────────────┐
│ 第一层：入口文档 (SKILL.md)                              │
│ 职责：导航中枢 + 全局变量 + 核心约束                     │
│ 字数目标：<500 字                                        │
│ 加载时机：技能触发时                                      │
├─────────────────────────────────────────────────────────┤
│ 第二层：阶段文档 (eval-*.md)                             │
│ 职责：定义阶段任务 + 任务目标 + 任务输出                  │
│ 字数目标：<2000 字                                       │
│ 加载时机：进入对应阶段时                                  │
├─────────────────────────────────────────────────────────┤
│ 第三层：流程文档 (processes/*.md)                        │
│ 职责：执行步骤 + 判断逻辑 + 命令示例                      │
│ 字数目标：<1500 字                                       │
│ 加载时机：执行具体流程时                                  │
└─────────────────────────────────────────────────────────┘
```

### 2.2 支撑层：参考文档

```
┌─────────────────────────────────────────────────────────┐
│ 支撑层：参考文档 (references/*.md)                       │
│ 职责：详细规范 + 接口说明 + 示例数据                      │
│ 字数目标：按需，无严格限制                                │
│ 加载时机：文档中明确指示时才加载                          │
└─────────────────────────────────────────────────────────┘
```

### 2.3 层级职责对比

| 层级 | 回答的问题 | 包含内容 | 不包含内容 |
|------|-----------|----------|-----------|
| **入口** | 这是什么？全局约束？ | 概述、阶段表、变量、核心 Red Flags | 详细流程、命令示例 |
| **阶段** | 做什么任务？目标？输出？ | 任务列表、目标、输出、阶段 Red Flags | 执行步骤、判断逻辑 |
| **流程** | 怎么执行？怎么判断？ | 步骤、判断表、命令、流程 Red Flags | 阶段概览、其他流程 |
| **参考** | 详细规范是什么？ | 接口说明、格式定义、示例数据 | 执行逻辑、判断条件 |

### 4.3 eval-set.md 拆分方案

**当前问题**：
- 4 个任务，其中任务2和任务3逻辑复杂
- 字数 ~2500，超标 25%

**拆分策略**：

#### 方案 A：任务下沉

将复杂任务的执行细节移至流程文档：

| 任务 | 当前内容 | 拆分后 |
|------|----------|--------|
| 任务1 | 获取评测集（有/无评测集分支） | 保留概要，分支细节移至 evalset-create.md |
| 任务2 | 解析与预处理（3 种状态分支） | 保留判断逻辑，子流程移至对应流程文档 |
| 任务3 | 标准化与评测点生成 | 保留概要，命令移至流程文档 |
| 任务4 | 上传评测集 | 保留命令（简单任务） |

**拆分后字数预估**：
- eval-set.md：~1500 字（精简 40%）
- 新增流程文档：无需新增，已有对应流程

#### 方案 B：阶段拆分

将"评测集处理"拆分为两个阶段：

| 新阶段 | 任务 | 字数预估 |
|--------|------|----------|
| 评测集准备 | 获取评测集、解析与预处理 | ~1200 字 |
| 评测集标准化 | 标准化、上传 | ~800 字 |

**问题**：增加了阶段数量，可能过度拆分

**推荐**：方案 A（任务下沉）

### 5.3 HARD-GATE 设计规范

**HARD-GATE 的作用**：
- 标注不可跳过的确认点
- 明确阻断条件
- 防止 Agent 绕过关键步骤

**HARD-GATE 模板**：

```markdown
<HARD-GATE>
{阻断条件}

**禁止行为**：
- {禁止行为1}
- {禁止行为2}

**必须行为**：
- {必须行为1}
</HARD-GATE>
```

**应用位置**：

| 流程 | HARD-GATE 位置 |
|------|----------------|
| evalset-parse.md | 字段映射确认前 |
| dimension-general.md | 维度权重确认前 |
| keypoint-process.md | 评测点预览确认前 |
| evalset-create.md | 问题集预览确认前 |

### 5.4 流程文档精简示例

**精简前**（keypoint-process.md 约 900 字）：

结构合理，但可以更简洁：

**精简后**：

```markdown
---
name: keypoint-process
description: Use when case-level evaluation requires generating keypoints
---

# 评测点生成流程

## 目标

根据问题、参考答案、上下文自动生成评测点。

核心原则：**必须调用模型生成，用户确认后方可保存**。

---

## 触发条件

定制用例级评测且标准化评测集无 keypoint 字段。

---

## 产物

| 文件 | 用途 |
|------|------|
| `evalset-standard.jsonl` | 添加 keypoint 字段后的标准化评测集 |

---

## 步骤

### 步骤1：场景与去重判断

**目的**：识别字段组合确定场景类型，处理多模型横评去重。

| case_id 状态 | 动作 |
|--------------|------|
| 存在重复值 | 按唯一 case_id 去重生成，复制到同 case_id 所有记录 |
| 无重复值 | 直接进入字段组合判断 |

**场景类型判断**（忽略 answer 字段）：

| 场景 | 包含字段 | 评估点来源 |
|------|----------|------------|
| Q | 仅 question | 基于问题推断 |
| QR | question + reference | 从参考答案提炼 |
| QC | question + context | 从上下文推导 |
| QRC | question + reference + context | 最佳效果 |

---

### 步骤2：生成评测点

**目的**：调用内置模型生成 1-3 个评测点。

| 数据量 | 执行方式 |
|--------|----------|
| ≤ 50 条 | 主 Agent 直接生成 |
| > 50 条 | 使用 Agent 工具分批执行 |

**生成方式**：调用 `{skill-dir}/scripts/utils/keypoint_prompts.py`

---

### 步骤3：预览与确认

<HARD-GATE>
**必须展示前 3 条数据并等待用户确认，不可自动继续。**
</HARD-GATE>

展示前 3 条评测点预览，询问用户：

| 操作 | 动作 |
|------|------|
| Y | → 步骤4 保存结果 |
| n | → 取消生成，不修改文件 |
| 调整 | → 按需求重新生成 |

---

### 步骤4：保存结果

**目的**：将评测点写入标准化评测集文件。

| 场景 | 动作 |
|------|------|
| 多模型横评 | 评测点复制到同 case_id 所有记录 |
| 单模型 | 直接写入各记录 |

---

## Red Flags

| 错误 | 原因 | 解决方案 |
|------|------|----------|
| 不调用模型直接编造评测点 | 质量无保障 | 必须调用内置模型生成 |
| 跳过用户确认直接保存 | 无法审核质量 | 必须展示前 3 条并等待确认 |
```

## 十、总结

### 10.1 核心改进点

| 改进维度 | 具体措施 | 预期效果 |
|----------|----------|----------|
| **职责分明** | 三层架构 + 职责边界定义 | 每个文档职责单一 |
| **Token 高效** | 入口精简 + 按需加载 | 节省 40%+ Token |
| **结构清晰** | 统一模板 + 层级限制 | 易于导航和理解 |
| **步骤明确** | HARD-GATE + 判断表格 | 执行无歧义 |
| **交互友好** | 关键点 HARD-GATE | 防止跳过确认 |
| **内容简洁** | 引用代替复制 + 细节下沉 | 无冗余重复 |

### 10.2 关键设计决策

1. **入口文档 <500 字**：牺牲完整性换取 Token 效率
2. **任务细节下沉**：阶段文档只定义"做什么"，流程文档定义"怎么做"
3. **HARD-GATE 标注**：用显式标注替代隐式约束
4. **流程编号规范化**：保持现有编号，按阶段分组展示；长期采用阶段前缀编号
