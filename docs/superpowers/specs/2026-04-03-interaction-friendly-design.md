# 模型评测 Skill 用户交互友好性改进设计

> **For agentic workers:** 本设计文档定义了进度追踪、路径记忆、分支提示、按需有向图四项改进。实现时需严格遵循格式规范。

**Goal:** 解决用户在使用 Skill 时的迷失感（不知道"我在哪"、"下一步做什么"）、选择困惑（不知道"该选哪个"、"选了会怎样"）、理解缺失（不知道"为什么要这样做"）三大痛点。

**优先级:** D（步骤嵌套迷失）> A（分支选择迷失）> B（状态判断迷失）> C（理解缺失）

---

## 问题分析

### 用户痛点识别

| 痛点 | 具体场景 | 用户困惑 |
|------|----------|----------|
| **D. 步骤嵌套迷失** | 任务下有步骤，步骤下有判断，判断后又有步骤 | "我刚才做了字段映射确认，现在是在哪个任务的哪个步骤？" |
| **A. 分支选择迷失** | 评测集处理步骤1询问"是否有评测集" | "选了'生成问题集'之后会发生什么？和'提供评测集'有什么不同？" |
| **B. 状态判断迷失** | 解析后检测答案字段状态，系统自动调用不同子流程 | "我的评测集部分有答案，为什么要问我确认类型？" |
| **C. 理解缺失** | 评测方式选择、字段映射确认等步骤 | "为什么要确认字段映射？不确认会怎样？" |

### 当前状态

| 维度 | 状态 | 说明 |
|------|------|------|
| 整体流程感知 | ✅ 已解决 | 流程入口展示四阶段概览 |
| 当前阶段感知 | ✅ 已解决 | 进入阶段时展示信息 |
| 当前任务感知 | ⚠️ 部分解决 | TodoWrite 显示任务，但步骤层级不清晰 |
| 当前步骤感知 | ❌ 待改进 | 步骤嵌套在任务内，用户看不到步骤进度 |
| 选择后流转感知 | ❌ 待改进 | 用户选择后，下一步是什么不明确 |
| 原因解释 | ❌ 待改进 | Why 说明缺失 |

---

## 设计方案

### 方案概述

采用**混合方案**，结合进度追踪、路径记忆、分支提示、按需有向图：

| 组件 | 解决痛点 | 输出时机 |
|------|----------|----------|
| **进度追踪 + 步骤列表** | D（步骤嵌套迷失） | 进入任务时 |
| **路径记忆 + 决策记录** | D、A（回溯上下文） | 进入步骤时（有分支） |
| **分支树形提示** | A、B（分支选择迷失） | 遇到分支判断时 |
| **按需有向图** | 用户主动了解流程 | 用户请求"详细"/"流程图"时 |

### 核心原则

| 原则 | 说明 |
|------|------|
| **渐进式披露** | 默认输出简化信息，完整信息按需展示 |
| **情境感知** | 根据用户行为动态调整输出详细程度 |
| **路径可回溯** | 展示"已走过的路径"帮助用户理解上下文 |

## 方案1: 进度追踪 + 步骤列表

### 核心目的

让用户一眼看到：
- 当前进度（做了多少，还剩多少）
- 步骤状态（已完成/当前/待执行）

### 输出时机

进入任务时输出。

### 格式模板

```
📍 {阶段} > {任务} [步骤 {当前}/{总数}]

┌─────────────────────────────────┐
│ ✓ 步骤1: {步骤名称}             │ ← 已完成
│ → 步骤2: {步骤名称}             │ ← 当前
│ ○ 步骤3: {步骤名称}             │ ← 待执行
│ ○ 步骤4: {步骤名称}（按需）     │ ← 条件执行
│ ○ 步骤5: {步骤名称}             │
└─────────────────────────────────┘
```

### 状态符号

| 符号 | 状态 | 说明 |
|------|------|------|
| `✓` | 已完成 | 该步骤已成功执行 |
| `→` | 当前 | 正在执行的步骤 |
| `○` | 待执行 | 尚未开始的步骤 |
| `~` | 已跳过 | 条件不满足跳过的步骤 |

### 示例

**进入评测集处理任务1时**：

```
📍 评测集处理 > 任务1 [步骤 1/5]

┌─────────────────────────────────┐
│ → 步骤1: 获取评测集             │
│ ○ 步骤2: 解析与状态处理         │
│ ○ 步骤3: 标准化                 │
│ ○ 步骤4: 评测点生成（按需）     │
│ ○ 步骤5: 上传评测集             │
└─────────────────────────────────┘
```

**步骤1完成后，进入步骤2时**：

```
📍 评测集处理 > 任务1 [步骤 2/5]

┌─────────────────────────────────┐
│ ✓ 步骤1: 获取评测集             │
│ → 步骤2: 解析与状态处理         │
│ ○ 步骤3: 标准化                 │
│ ○ 步骤4: 评测点生成（按需）     │
│ ○ 步骤5: 上传评测集             │
└─────────────────────────────────┘
```

## 方案2: 路径记忆 + 决策记录

### 核心目的

让用户知道：
- 之前做了什么选择
- 当前在做什么
- 接下来会发生什么

### 输出时机

进入有分支的步骤时输出。

### 格式模板

```
📍 {阶段} > {任务} > 步骤{n}: {步骤名称}

已走路径：
  {步骤n-1}: {名称} → 选择"{用户选择}"

当前判断：{判断条件}
  ├─ {选项A} → {后果A}
  ├─ {选项B} → {后果B}
  └─ {选项C} → {后果C}
```

### 示例

**进入步骤2（解析与状态处理）时**：

```
📍 评测集处理 > 任务1 > 步骤2: 解析与状态处理

已走路径：
  步骤1: 获取评测集 → 选择"已有评测集"

当前判断：检测答案字段状态
  ├─ 答案全空 → 进入模型选择流程
  ├─ 答案全有 → 进入字段校验流程
  └─ 部分有答案 → 需确认评测集类型
```

**判断结果出来后（部分有答案）**：

```
📋 判断结果：部分有答案

请确认评测集类型：
  1. 只有问题 → 进入模型选择流程
  2. 问题+答案 → 进入字段校验流程
```

### 路径记录格式

| 格式 | 说明 |
|------|------|
| `步骤n: 名称 → 选择"X"` | 用户主动选择 |
| `步骤n: 名称 → 结果X` | 系统自动判断 |
| `步骤n: 名称 → 跳过` | 条件不满足跳过 |

## 方案3: 分支树形提示

### 核心目的

在用户需要做选择时，清晰展示各选项的后果。

### 输出时机

遇到需要用户选择的分支点时输出。

### 格式模板

```
{判断/选择问题}
  ├─ {选项A} → {后果A}
  ├─ {选项B} → {后果B}
  └─ {选项C} → {后果C}
```

### 示例

**评测集来源选择**：

```
是否有现成的评测集？
  ├─ 已有评测集 → 直接解析评测集
  ├─ 无评测集 → 生成问题集后解析
  └─ 需要帮助 → 引导选择
```

**评测方式选择**：

```
请选择评测方式：
  ├─ 通用维度级 → 宏观全面覆盖，适用标准评测
  └─ 定制用例级 → 细节偏好对齐，适用定制需求
```

## 方案4: 按需有向图

### 核心目的

用户主动请求时，输出完整的流程有向图。

### 触发方式

| 用户输入 | 系统响应 |
|----------|----------|
| "详细" / "流程图" / "展开" | 输出当前层级的完整有向图 |
| "简洁" / "收起" | 切换回简化模式 |

### 有向图格式规范

**符号说明**：

| 符号 | 用途 |
|------|------|
| `[步骤]` | 普通节点 |
| `{判断?}` | 分支节点 |
| `[[流程]]` | 流程引用 |
| `-->` | 连接箭头 |
| `◀` | 当前位置标记 |
| `⚠️` | 需确认标记 |

**示例**（用户请求"流程图"时）：

```
任务1: 处理评测集 - 完整流程图

[步骤1: 获取评测集] --> {是否有评测集?}
                          │
                          ├── 已有评测集 ──→ [步骤2: 解析与状态处理]
                          │
                          ├── 无评测集 ──→ [[流程1: 问题集生成]] ──→ [步骤2]
                          │
                          └── 无法判断 ──→ [询问用户] ──→ {是否有评测集?}

[步骤2] --> {检测答案状态}
                │
                ├── 答案全空 ──→ [[流程7]] ──┐
                │                              │
                ├── 答案全有 ──→ [[流程8]] ──┤
                │                              │
                └── 部分有答案 ──→ [询问类型] ──┴─→ [步骤3: 标准化]

当前位置：[步骤2: 解析与状态处理] ◀
```

## 输出顺序规范

| 时机 | 输出内容 | 说明 |
|------|----------|------|
| **进入阶段** | 阶段名称 + 任务列表 | 了解阶段构成 |
| **进入任务** | 进度追踪 + 步骤列表 | 了解任务进度 |
| **进入步骤（有分支）** | 路径记忆 + 决策记录 + 分支提示 | 了解上下文和选择 |
| **进入步骤（无分支）** | 当前位置行 | 简化输出 |
| **遇到分支判断** | 判断结果 + 后续动作 | 了解系统决策 |
| **遇到用户选择** | 分支树形提示 | 了解选项后果 |
| **用户请求"详细"** | 完整有向图 | 深入了解流程 |

### 完整示例流程

**1. 进入评测集处理阶段**：

```
📍 当前位置：评测集处理阶段

评测集处理阶段包含 1 个任务：
  → 任务1: 处理评测集
```

**2. 进入任务1**：

```
📍 评测集处理 > 任务1 [步骤 1/5]

┌─────────────────────────────────┐
│ → 步骤1: 获取评测集             │
│ ○ 步骤2: 解析与状态处理         │
│ ○ 步骤3: 标准化                 │
│ ○ 步骤4: 评测点生成（按需）     │
│ ○ 步骤5: 上传评测集             │
└─────────────────────────────────┘
```

**3. 进入步骤1（有分支）**：

```
📍 评测集处理 > 任务1 > 步骤1: 获取评测集

当前判断：是否有评测集？
  ├─ 已有评测集 → 直接解析评测集
  ├─ 无评测集 → 生成问题集后解析
  └─ 无法判断 → 询问用户

请问您是否有现成的评测集？
```

**4. 用户选择"已有评测集"后，进入步骤2**：

```
📍 评测集处理 > 任务1 [步骤 2/5]

┌─────────────────────────────────┐
│ ✓ 步骤1: 获取评测集             │
│ → 步骤2: 解析与状态处理         │
│ ○ 步骤3: 标准化                 │
│ ○ 步骤4: 评测点生成（按需）     │
│ ○ 步骤5: 上传评测集             │
└─────────────────────────────────┘

已走路径：
  步骤1: 获取评测集 → 选择"已有评测集"

当前判断：检测答案字段状态
  ├─ 答案全空 → 进入模型选择流程
  ├─ 答案全有 → 进入字段校验流程
  └─ 部分有答案 → 需确认评测集类型
```

**5. 判断结果出来后**：

```
📋 判断结果：答案全有 → 进入字段校验流程
```

## Why 说明（后续迭代）

Why 说明不在本次实现范围内，后续迭代时遵循以下原则：

| 原则 | 说明 |
|------|------|
| 优先级最低 | 先验证方案1+方案2效果，再考虑添加 Why 说明 |
| 聚焦高频困惑点 | 先在 3-5 个最容易产生困惑的步骤添加 |
| 简短控制在 50 字内 | 避免 token 过载 |
| 使用 💡 符号 | 区分于其他标记 |

## 验证标准

### 功能验证

| 验证项 | 验证方法 |
|--------|----------|
| 进度追踪正确输出 | 检查 `[步骤 n/m]` 格式和状态符号 |
| 路径记忆正确记录 | 检查"已走路径"是否包含之前的选择 |
| 分支提示清晰 | 检查树形结构是否正确展示选项和后果 |
| 按需有向图触发正确 | 检查用户请求时是否输出完整图 |

### Token 效率验证

| 验证项 | 目标值 |
|--------|--------|
| 进入任务时输出 | < 150 token |
| 进入步骤时输出 | < 100 token |
| 分支提示输出 | < 80 token |
| 完整有向图输出 | < 300 token |

### 用户体验验证

| 验证项 | 验证方法 |
|--------|----------|
| 解决迷失感 | 用户能回答"我在哪"、"进度如何" |
| 解决选择困惑 | 用户能理解"选了会怎样" |
| 不增加认知负担 | 默认输出简洁，详细信息按需展示 |

### 格式模板

**进入阶段时**：
```
📍 当前位置：初始化阶段
```

**进入任务时**：
```
📍 当前位置：初始化 > 任务1：环境及依赖检测
```

**进入步骤时**：
```
📍 当前位置：初始化 > 任务1 > 步骤3：执行环境检测
```

**完成步骤时**（可选，仅在关键节点输出）：
```
📍 进度更新：✅ 步骤3 完成 → 进入步骤4
```

### 设计原则

| 原则 | 说明 |
|------|------|
| 极简定位 | 仅输出当前层级位置，不展示完整列表 |
| 有向图为主 | 完整流程、分支路径由有向图展示 |
| 关键节点更新 | 仅在完成任务或重要分支时输出进度更新 |

### 与现有规范的整合

| 原有方式 | 新方式 | 原因 |
|----------|--------|------|
| TodoWrite 工具调用 | 路径追踪文本输出 | 简化输出，突出有向图 |
| 流程入口展示（四阶段概览） | **替换为阶段级有向图** | 更直观展示四阶段流转 |

## 方案2: 流程有向图设计

### 核心目的

在进入阶段/任务/步骤/流程时，展示完整的分支路径有向图，让用户一眼看到：
- 有哪些分支选择
- 每个选择后会走向哪里
- 当前处于哪个节点

### 展示时机与层级

| 层级 | 展示时机 | 展示内容 | 目的 |
|------|----------|----------|------|
| **阶段级** | 进入阶段时 | 该阶段下的所有任务 | 了解阶段任务构成 |
| **任务级** | 进入任务时 | 该任务下的所有步骤 | 了解任务步骤构成 |
| **步骤级** | 进入步骤时 | 该步骤内的判断分支 | 了解分支细节 |
| **流程级** | 执行独立流程时 | 流程内所有步骤 | 了解流程全貌 |

### 展示原则

| 原则 | 说明 |
|------|------|
| **只展开一层** | 每层级有向图仅展示下一级内容，不深入展开 |
| **进入时输出一次** | 进入该层级时输出有向图，后续不再重复输出 |
| **位置回溯即可** | 用户通过当前位置标识，向上回溯即可定位 |
| **同级不重复** | 进入步骤后，仅输出步骤级有向图，不再输出任务级图 |

### ASCII 有向图格式规范

使用 ASCII 字符绘制，确保在终端环境正常显示：

| 元素 | ASCII 表示 | 说明 |
|------|------------|------|
| 普通节点 | `[步骤1]` | 方括号包围 |
| 分支节点 | `{判断?}` | 花括号表示判断点 |
| 流程引用 | `[[流程1]]` | 双层方括号表示调用子流程 |
| 箭头 | `-->` | 单向连接 |
| 分支标注 | `-->|条件|` | 箭头上标注条件 |
| 当前节点 | `◀ 当前` | 节点后标记当前位置 |

### 绘制规范

| 规范 | 说明 |
|------|------|
| **无外框** | 不使用 `┌─┐` 方框，避免中英文混用导致对齐问题 |
| **纯文本** | 使用代码块格式，让终端自动换行 |
| **中文优先** | 所有节点和条件使用中文描述，避免技术术语 |
| **缩进对齐** | 使用空格缩进，同层级节点对齐 |
| **从上到下** | 主流程从上到下排列，分支向右延伸 |
| **无产物节点** | 有向图仅展示流程步骤，不展示输出文件（如 env.cfg） |

---

### 阶段级有向图示例

**展示时机**：进入阶段时（仅输出一次）

```
初始化阶段

[任务1: 环境检测] --> [任务2: 鉴权验证] --> [任务3: 会话目录]

当前位置：[任务1: 环境检测] ◀
```

---

### 任务级有向图示例

**示例1：初始化阶段 - 任务1**

```
任务1: 环境检测

[步骤1: 检查缓存] --> {缓存存在?}
                          │
                          ├── 是 ──→ [步骤2: 验证缓存] --> {缓存有效?}
                          │                                      │
                          │                                      ├── 是 ──→ 完成任务
                          │                                      │
                          │                                      └── 否 ──┐
                          │                                               │
                          └── 否 ──────────────────────────────────────────┤
                                                                           │
                                                                           ↓
                                                                    [步骤3: 执行检测] --> [步骤4: 生成缓存] --> 完成任务
```

**说明**：
- 缓存存在且有效 → 直接完成任务
- 缓存不存在 或 缓存无效 → 两条路径汇聚到步骤3

**示例2：评测集处理阶段 - 任务1**

```

                │                           │                   │
                │                           └── Y ──→ [Y处理] ──┤
                │                                               │
                └── B ──→ [处理B] ──┐                          │
                                     │                          │
                                     ↓                          ↓
                              [共同后续步骤] ◀──────────────────┘
```

**符号说明**：

| 符号 | 用途 | 示例 |
|------|------|------|
| `--→` | 主流程箭头 | `[步骤1] --→ [步骤2]` |
| `──→` | 分支箭头（全角） | `├── 条件 ──→ [分支]` |
| `──┐` | 分支向下汇聚起点 | `├── A ──→ [X] ──┐` |
| `──┴─→` | 分支汇聚连接 | `└── B ──→ [Y] ──┴─→ [后续]` |
| `│` | 垂直连接线 | 连接上下节点 |
| `├` | 分支起点 | 表示分支开始 |
| `└` | 最后分支 | 表示最后一个分支 |
| `↓` | 向下箭头 | 明确流向 |
| `◀` | 当前位置标记 | `[步骤1] ◀` |
| `[[]]` | 流程引用 | `[[流程1: 问题集生成]]` |
| `{}` | 判断节点 | `{有评测集?}` |
| `⚠️` | 需确认标记 | `[步骤] ⚠️ 需用户确认` |
| `（跳过）` | 条件跳转标注 | `[步骤2]（跳过）` |

### 流程结构规范

**结构1：分支汇聚**

多个分支路径汇聚到同一节点时，使用右侧汇聚语法：

```
{条件判断}
       │
       ├── 条件A ──→ [处理A] ──┐
       │                       │
       └── 条件B ──→ [处理B] ──┴─→ [共同后续步骤]
```

**结构2：条件跳转**

满足条件时跳过某些步骤：

```
[步骤1] --> {需要步骤2?}
                  │
                  ├── 是 ──→ [步骤2] --> [步骤3]
                  │
                  └── 否 ──→ [步骤3]（跳过步骤2）
```

**结构3：条件跳转（简化表示）**

在箭头上直接标注跳转条件：

```
[步骤1] -- 满足条件 --> [步骤2]
    │
    └-- 不满足条件 --（跳过）--> [步骤3]
```

## 方案2: 流转预告机制设计

### 核心原则

| 原则 | 说明 |
|------|------|
| 预告在先 | 在用户做出选择前，说明每个选项的后续流程 |
| 回顾在后 | 在系统做出判断后，说明判断结果和后续流程 |
| 有向图优先 | 有向图已展示时，预告简化为一句话 |
| 简短聚焦 | 预告/回顾内容控制在 50 字以内 |

### 预告格式

**有向图已展示时（默认）**：
```
⚠️ 选择预告：参考上方流程图，选择后进入对应分支路径
```

**无有向图时（完整格式）**：
```
⚠️ 选择预告：
- 选择「{选项A}」→ {流程A简述}
- 选择「{选项B}」→ {流程B简述}
```

### 回顾格式

```
📋 判断结果：{value} → {后续处理简述}
```

**示例**：
```
📋 判断结果：status = partial → 询问用户确认评测集类型后调用对应流程
```
