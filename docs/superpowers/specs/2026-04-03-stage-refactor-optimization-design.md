# stage-refactor 技能优化设计 v2

## 概述

基于 superpowers 技能规范、Agent Skill 五种设计模式及 model-evaluation 阶段文档实践，优化 `stage-refactor` 技能，提升 Token 效率、消除内容冗余。

---

## 问题分析

### 当前问题（对照 superpowers 规范）

| 类别 | 问题 | 对照依据 | 影响 |
|------|------|----------|------|
| Description | "阶段文档重构专家"总结功能 | Writing-Skills规范："Description = 触发条件，绝不总结工作流程" | 触发判断模糊 |
| 字数冗余 | ~248行，五维度报告模板过长 | systematic-debugging：主文档<200行 | Token 消耗高 |
| 内容重复 | Red Flags + 常见错误内容重叠 | eval-execute.md：两者互补不重复 | 信息冗余 |
| 判断依据冗余 | 重复 writing-guide.md 内容 | Writing-Skills规范："引用不重复" | Token 冗余 |
| 流程冗余 | 四阶段中第二阶段可合并 | 流水线模式：门控明确即可 | 流程复杂 |

### 对照 model-evaluation 阶段文档实践

| 文档 | Red Flags + 常见错误处理 | 字数控制 |
|------|--------------------------|----------|
| eval-init.md | 分开保留，内容互补 | ~250行 |
| eval-build.md | 分开保留，内容互补 | ~225行 |
| eval-execute.md | 合并"常见借口"到 Red Flags | ~220行 |

**发现**：stage-refactor 的 Red Flags 与常见错误存在实质性重复（如"跳过用户确认"），不同于阶段文档的互补设计。

---

## 设计方案

### 核心优化

| 变更项 | 变更内容 | 理由 |
|--------|----------|------|
| 字数目标 | 从 ~248 行精简至 ~100 行 | Token 效率优先 |
| 执行流程 | 三阶段：分析→执行→验证 | 合并冗余阶段 |
| 五维度报告 | 清单式输出，不输出详细表格 | 借鉴 systematic-debugging |
| 检查清单 | 引用 writing-guide.md，不内嵌 | 引用不重复原则 |
| 判断依据章节 | 删除，改引用 writing-guide.md | 消除冗余 |
| Red Flags + 常见错误 | 合并为"禁止与错误速查" | 去重整合 |

### 目录结构

```
.claude/skills/stage-refactor/
├── SKILL.md                    # 主技能文件（~100行）
└── references/
    └── writing-guide.md        # 阶段文档书写规范（已有，复用）
```

**决策**：不新增 checklist.md，复用已有 writing-guide.md。

## 详细设计

### 1. YAML Frontmatter

```yaml
---
name: stage-refactor
description: Use when user needs to adjust, optimize, or refactor eval-init.md, eval-build.md, eval-set.md, or eval-execute.md stage documents
metadata:
  pattern: reviewer+pipeline
  domain: stage-refactor
  steps: "3"
---
```

**改进点**：
- Description 仅包含触发条件
- metadata 标注审查器+流水线组合模式

### 2. SKILL.md 正文结构

| 章节 | 内容 | 行数 |
|------|------|------|
| 标题描述 | 简短陈述 | ~2行 |
| 核心原则 | 表格形式（4项） | ~8行 |
| 执行流程 | 三阶段带门控条件 | ~40行 |
| 五维度检查 | 清单表格（引用规范） | ~15行 |
| 禁止与错误速查 | 合并表格 | ~20行 |
| 协同说明 | 与 process-refactor 协同 | ~8行 |

**总计**：~100行

### 3. 执行流程设计

#### 第一阶段：分析现状

```
1. 读取目标阶段文档
2. 读取 references/writing-guide.md
3. 对照规范，输出差异清单（五维度）
4. ⚠️ 询问："是否继续调整？"（不得进入第二阶段直到用户确认）
```

**差异清单格式**（清单式，非详细表格）：

```
维度1-结构格式：{通过/缺失}项：YAML、目标、前置验证...
维度2-交互友好：{问题}项：缺少确认标注...
维度3-内容精简：{问题}项：示例过多...
维度4-逻辑合理：{问题}项：任务顺序待确认...
维度5-阶段特有：{缺失}项：流程速查表...
```

#### 第二阶段：执行调整

```
用户确认后执行：
- 结构调整：补充缺失章节
- 格式修正：YAML、表格格式
- 交互优化：添加 ⚠️ 标注
- 内容精简：列表改表格
- ⚠️ 逻辑调整：需单独确认

每项完成后：✓ {调整项}
⚠️ 全部完成后询问："是否进入验证？"（不得进入第三阶段直到用户确认）
```

**门控条件**：
- 逻辑调整（任务顺序、步骤合并）必须单独确认
- 执行完毕必须确认后才能进入验证

#### 第三阶段：验证交付

```
1. 读取调整后文档
2. 执行五维度快速检查
3. 未通过项 → 返回第二阶段
4. 全部通过 → 输出调整摘要
5. ⚠️ 询问："是否保存？"（不得结束直到用户确认）
```

### 4. 五维度检查清单

| 维度 | 检查项 |
|------|--------|
| 结构格式 | YAML、目标、前置验证、完成标志、流程速查、任务层级、Red Flags、常见错误、变量速查 |
| 交互友好 | 确认点标注 ⚠️、目的说明、选项清晰、状态表完整 |
| 内容精简 | 示例数量合理、无重复内容、表格优先、无过度解释 |
| 逻辑合理 | 任务顺序合理、步骤粒度适中、判断条件完整、命令参数正确、业务约束体现 |
| 阶段特有 | 前置验证完整、完成标志顺序正确、流程速查表完整、任务→步骤层级清晰 |

**详细标准见 references/writing-guide.md**

### 5. 禁止与错误速查表格

合并 Red Flags 与常见错误，去重整合：

| 类型 | 内容 | 处理 |
|------|------|------|
| **禁止** | 跳过用户确认阶段 | 必须等待确认 |
| **禁止** | 未读取规范就执行 | 先加载 writing-guide.md |
| **禁止** | 未经确认修改逻辑调整 | 先询问用户 |
| **禁止** | 删除必需章节 | 保留目标、完成标志、Red Flags |
| **禁止** | 调整后未执行验证 | 必须执行五维度检查 |
| **借口** | "规范已熟悉无需加载" | 必须重新加载规范 |
| **借口** | "调整简单无需确认" | 所有调整需确认 |
| **错误** | 字数超限（>2000字） | 精简示例和解释 |
| **错误** | 表格格式错乱 | 检查对齐和列数 |
| **错误** | 步骤顺序混乱 | 对照规范重新排序 |
| **错误** | 流程速查表缺失 | 补充子流程引用汇总 |

### 6. 协同说明

| 技能 | 目标文档 | 允许逻辑调整 |
|------|----------|--------------|
| process-refactor | processes/*.md 流程文档 | ❌ 禁止 |
| stage-refactor | eval-*.md 阶段文档 | ✅ 允许（需用户确认） |

**协同场景**：阶段重构发现子流程问题 → 先完成阶段重构 → 再调用 process-refactor
