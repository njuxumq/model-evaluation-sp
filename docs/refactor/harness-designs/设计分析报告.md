# 重构保障设计分析报告

## 一、分析背景

本报告基于 superpowers 插件的设计思想，对 harness-designs 目录下的重构保障设计进行深入分析。

**分析文档**：
- 重构保障大纲.md
- 保障详细设计.md
- 实施详细步骤.md

**参考标准**：
- superpowers/writing-skills/SKILL.md（Skill 编写规范）
- superpowers/writing-plans/SKILL.md（计划编写规范）
- superpowers/executing-plans/SKILL.md（计划执行规范）
- superpowers/subagent-driven-development/SKILL.md（子代理驱动开发）
- superpowers/test-driven-development/SKILL.md（TDD 规范）
- superpowers/verification-before-completion/SKILL.md（验证规范）

---

## 二、核心理念对比

### 2.1 马具工程思想

**设计意图**：在执行重构前，先构建一套"马具"（流程和工具）来约束和引导重构执行。

**评价**：这个理念本身是正确的。superpowers 也强调"先约束后执行"的原则，例如 TDD 的"NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST"。

**但存在关键差异**：
| 对比项 | superpowers | harness-designs |
|--------|-------------|-----------------|
| 约束来源 | 外部验证（测试命令、子代理审查） | 内部验证（JSON 产物门控） |
| 验证方式 | 可执行的命令（pytest、npm test） | Agent 自我检查 JSON 字段 |
| 失败可见性 | 测试失败有明确输出 | 门控失败可能被 Agent 忽略 |

---

## 三、发现的核心问题

### 问题1：缺少 TDD 验证流程

**superpowers 铁律**：
```
NO SKILL WITHOUT A FAILING TEST FIRST
```

**harness-designs 现状**：
- 设计文档直接编写，没有经过压力场景测试
- 没有记录"Agent 在没有这个设计时会犯什么错误"
- 没有 RED-GREEN-REFACTOR 的迭代过程

**风险**：
- 设计可能解决了错误的问题
- 设计可能有漏洞，Agent 可以绕过约束
- 设计可能过于复杂，Agent 难以执行

**建议**：
1. 编写压力场景（如"重构时遗漏依赖文件"、"执行顺序错误"）
2. 让 Agent 在没有设计的情况下执行重构，记录失败模式
3. 根据失败模式修订设计
4. 重新测试直到 Agent 遵循设计

---

### 问题2：门控条件的自我参照问题

**设计描述**：
```json
{
  "completeness_check": {
    "all_files_identified": true,
    "all_deps_traced": true,
    "all_risks_assessed": true
  }
}
```

**核心问题**：Agent 如何知道自己"遗漏了文件"？

这是一个**自我参照悖论**：
- 门控条件要求 Agent 确认"所有文件已识别"
- 但如果 Agent 不知道某个文件存在，它不会知道自己遗漏了
- 门控变成了形式主义的"打勾"而非真正的验证

**superpowers 的解决方案**：
- 使用**外部验证**：运行测试命令，如果失败说明遗漏
- 使用**子代理审查**：另一个 Agent 独立检查
- 使用**明确的标准**：`ls processes/*.md | wc -l` 应输出 3

**建议**：
1. 门控条件应该是**可机械化验证**的命令
2. 避免"所有 X 已完成"这类主观判断
3. 使用具体数量或明确的存在性检查

**改进示例**：
```markdown
# 不好的门控
- all_files_identified: true

# 好的门控
- 执行 `git diff --name-only HEAD~5` 获取变更文件列表
- 执行 `grep -r "引用" --include="*.md" | grep -v "node_modules"` 验证引用完整性
- 对比两个列表，确认无遗漏
```

---

### 问题3：验证方法的模糊性

**设计中的验证描述**：
| 验证项 | 验证方法 | 通过条件 |
|--------|----------|----------|
| 新流程可执行 | 模拟流程执行 | 无阻塞错误 |
| 原有场景正常 | 回归测试 | 原有场景正常 |

**问题分析**：
1. **"模拟流程执行"是什么？**
   - 是否要实际调用评测 API？
   - 是否要实际上传评测集？
   - 如果不调用真实服务，如何验证"可执行"？

2. **"无阻塞错误"如何定义？**
   - 什么算"阻塞"？
   - 警告信息算不算阻塞？
   - Agent 如何判断自己是否"阻塞"？

3. **"原有场景正常"如何验证？**
   - 没有定义测试套件
   - 没有具体的测试命令
   - 依赖于 Agent 的主观判断

**superpowers 的解决方案**：
```markdown
# 明确的验证命令
Run: `pytest tests/path/test.py::test_name -v`
Expected: PASS
```

**建议**：
1. 定义具体的验证命令
2. 定义期望的输出格式
3. 对于 skill 重构，需要定义：
   - 如何启动一个评测会话
   - 如何执行每个阶段
   - 如何判断阶段完成

---

### 问题4：缺少 Red Flags 和 Rationalization 表格

**superpowers 的特色设计**：
```markdown
| Excuse | Reality |
|--------|---------|
| "Too simple to test" | Simple code breaks. Test takes 30 seconds. |
| "I'll test after" | Tests passing immediately prove nothing. |
```

**harness-designs 现状**：
- 没有预见 Agent 可能的理性化借口
- 没有提供"Red Flags"列表供 Agent 自检
- 依赖于 Agent 的"自觉执行"

**风险**：
- Agent 在时间压力下会寻找借口跳过步骤
- "这个门控可以跳过"
- "这个验证不适用于我的情况"
- "我已经确认了，不需要再检查"

**建议**：
添加 Red Flags 和 Rationalization 表格：

```markdown
## Red Flags - STOP

- 跳过分析直接开始重构
- "我已经知道要改哪些文件"
- 门控条件标记为 true 但没有验证过程
- "这个依赖不重要"
- 执行顺序与设计不符

## 常见理性化借口

| 借口 | 现实 |
|------|------|
| "这个重构很简单" | 简单重构也会引入 bug。遵循流程。 |
| "我跳过门控是为了效率" | 跳过验证 = 跳过质量保证。 |
| "不需要回滚方案" | 每个高风险操作都需要回滚方案。 |
| "测试环境不同于生产" | 那更需要严格验证。 |
```

---

### 问题5：缺少子代理架构

**superpowers 的核心设计**：
```
Fresh subagent per task + two-stage review (spec then quality)
```

**harness-designs 的设计**：
- 单一 Agent 执行所有步骤
- 没有"独立审查者"角色
- 审核器是同一个 Agent 的"自我审查"

**问题**：
- Agent 无法发现自己的盲点
- "自我审查"容易流于形式
- 没有真正的质量门控

**建议**：
1. 考虑引入子代理进行独立审查
2. 或者使用 `superpowers:subagent-driven-development` 作为执行框架
3. 至少定义"审查检查清单"让审查过程可追溯

---

### 问题6：中间产物（JSON）的问题

**设计中的 JSON 产物**：
- scope.json
- design.json
- review.json
- changes.json
- validation.json

**问题分析**：

1. **格式正确性验证**：
   - Agent 生成的 JSON 可能有语法错误
   - 设计中没有"JSON 格式验证"步骤

2. **一致性维护**：
   - 五个 JSON 文件之间可能有引用关系
   - 如果 scope.json 更新，design.json 是否需要更新？
   - 设计没有定义版本一致性机制

3. **可追溯性**：
   - JSON 文件没有版本控制建议
   - 无法回溯"为什么做出这个决定"

**建议**：
1. 简化中间产物，考虑合并为单一状态文件
2. 或者使用 git commit 作为每个阶段的检查点
3. 添加 JSON schema 验证步骤

---

### 问题7：实施步骤的依赖关系问题

**实施步骤设计**：
| 阶段 | 任务 | 依赖 |
|------|------|------|
| 1 | 创建重构保障流程 | 无 |
| 2 | 创建编排层 | 阶段1 |
| ... | ... | ... |
| 10 | 最终校验 | 阶段1-9 |

**问题分析**：

1. **阶段依赖假设**：
   - 设计假设阶段必须按顺序执行
   - 但某些阶段可能可以并行（如阶段3知识层和阶段4审查层）

2. **缺少依赖图**：
   - 设计没有可视化依赖关系
   - Agent 可能不清楚哪些阶段可以并行

3. **回滚粒度**：
   - 设计说"阶段1-6用 git checkout"
   - 但如果阶段4失败，是否需要回滚阶段3？

**建议**：
1. 添加依赖关系图（使用 DOT 格式）
2. 明确哪些阶段可以并行
3. 定义每个阶段的回滚粒度

## 四、与 superpowers 设计原则的对比

### 4.1 CSO（Claude Search Optimization）原则

**superpowers 要求**：
- Description = 触发条件，不总结工作流程
- 避免让 Agent 只读 description 就跳过 skill 内容

**harness-designs 现状**：
- 没有明确定义 YAML frontmatter
- 没有区分"触发条件"和"工作流程"

### 4.2 Token 效率原则

**superpowers 要求**：
- 高频加载的 skill < 200 words
- 详细内容放在单独文件中按需加载

**harness-designs 现状**：
- 三份文档合计约 27,000 字符
- 可能需要进一步精简或拆分

### 4.3 验证驱动原则

**superpowers 要求**：
- 每个声明都要有验证命令
- "Evidence before claims, always"

**harness-designs 现状**：
- 验证方法描述模糊
- 缺少具体的验证命令

## 五、改进建议总结

### 5.1 短期改进（保持现有框架）

1. **添加 Red Flags 和 Rationalization 表格**
   - 预见 Agent 可能的借口
   - 提供明确的"停止条件"

2. **明确验证方法**
   - 将模糊的验证条件改为可执行的命令
   - 定义期望的输出格式

3. **添加 YAML frontmatter**
   - 遵循 CSO 原则
   - Description 只描述触发条件

4. **简化中间产物**
   - 考虑合并 JSON 文件
   - 或使用 git commit 作为状态检查点

### 5.2 中期改进（引入 superpowers 机制）

1. **引入子代理审查**
   - 使用 `superpowers:subagent-driven-development` 框架
   - 让独立子代理执行审查

2. **TDD 验证流程**
   - 编写压力场景测试设计
   - 记录失败模式并修订设计

3. **与现有 superpowers skills 集成**
   - 使用 `superpowers:verification-before-completion`
   - 使用 `superpowers:systematic-debugging` 处理失败

### 5.3 长期改进（重新设计）

1. **考虑直接使用 superpowers 作为执行框架**
   - harness-designs 只定义"重构内容"
   - 执行流程由 superpowers skills 控制

2. **建立回归测试套件**
   - 定义"原有场景正常"的测试命令
   - 每次重构后自动运行测试

---

## 六、结论

harness-designs 的设计理念（马具工程）是正确的，但在实现层面存在以下核心问题：

1. **缺少 TDD 验证**：设计本身没有被测试验证过
2. **自我参照的门控**：Agent 难以验证自己的盲点
3. **验证方法模糊**：缺少可执行的验证命令
4. **缺少 Red Flags**：没有预见 Agent 的理性化借口
5. **缺少子代理架构**：没有独立审查机制

**核心建议**：
- 如果要继续使用当前设计，需要至少添加 Red Flags 表格和明确的验证命令
- 如果追求更高的可靠性，建议与 superpowers 框架集成，使用子代理驱动开发模式
