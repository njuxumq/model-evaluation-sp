# 评测 Skill 研发提效实践

## 1. 研发背景

星评测 2.0 平台为用户提供 AI 模型评测服务，但传统前端操作模式存在明显门槛：

> 用户在前端创建评测任务时，需要：第一步选择评测集（需理解什么是评测集、JSONL/CSV/XLSX 格式差异）；第二步选择评委模型（需理解模型评分机制、成本差异）；第三步配置评测维度（需从 46 个内置维度中选择、理解权重含义、判断是否需要定制化）...出错时，用户只能翻阅技术文档或求助客服。

评测 Skill 的目标是将这一流程迁移至 Claude Code 终端对话环境，通过自然语言交互引导用户完成评测任务构建，屏蔽专业知识门槛，实现"几轮对话即可提交评测任务"的体验。

Skill 承担两项职责：对话引导（根据用户描述的场景，主动推荐配置方案）和任务执行（提交任务、监控状态、展示结果）。

开发面临的核心挑战：需对接多个后端服务接口、正确管理四阶段状态流转、妥善处理 Token 失效等异常场景、平衡引导性与灵活性。

## 2. 三阶段踩坑之旅

### 2.1 第一阶段：指令过载的困境（15000 字指南的反噬）

开发初期，团队编写了一份 15000 字的详尽"评测 Skill 构建指南"，涵盖后端接口说明、评测流程设计、异常处理规则、脚本设计等内容，期望为 Claude Code 提供充分的背景知识。

**问题表现**：生成的 Skill 流程僵硬——用户已提供部分信息，Skill 仍按预设流程逐一询问；无法根据对话上下文判断用户意图；遇到非预期情况时僵化执行预设规则。

**根本原因**：Skill 文档描述的是"怎么做"而非"做什么"。过度规定步骤细节，导致 Claude Code 被困在预设流程中。详尽的异常处理规则反而限制了灵活性。

**转折契机**：团队阅读了 skill-creator 和社区其他优秀 Skill 的源码，发现它们的文档普遍精简（1000-3000 字），聚焦于"目标"而非"步骤"。这触发了关键认知转变——Claude Code 本身具备推理能力，不需要我们告诉它每一步该怎么走。

**核心认知**：Skill 应描述"做什么"、"目标是什么"，至于"怎么做"，让 Claude Code 根据对话上下文自主判断。

---

### 2.2 第二阶段：精简指令的突破（3000 字指南的成效）

基于反思，团队将构建指南精简至 3000 字。精简的核心逻辑是：

> Claude Code 本身具备推理能力，不需要我们告诉它每一步该怎么走。我们只需要告诉它：阶段目标是什么、完成标志是什么、遇到问题时可以参考哪些文档。剩下的，让它根据对话上下文自主判断。

具体做法：聚焦于四阶段流程框架、阶段完成标志、核心变量定义、脚本功能定位；删除详细步骤分解、异常处理规则、对话交互模板；将完整接口定义移至 references 按需加载。

**效果对比**：

| 维度 | 第一版 | 第二版 |
|------|--------|--------|
| 交互流畅度 | 机械式按步骤询问 | 根据上下文灵活引导 |
| 流程跳转能力 | 无法跳过已完成步骤 | 可根据用户意图跳转 |
| 异常处理 | 僵化执行预设规则 | 根据实际情况灵活应对 |
| Token 消耗 | 高 | 低 |

精简指南效果更好的原因：重点突出、自主空间大、Token 效率高。Claude Code 能清晰理解核心目标，不被细节淹没。

---

### 2.3 第三阶段：工程化迭代（gsd 驱动的规范化实践）

第二阶段生成的 Skill 已具备可用性，但后续迭代面临问题：需求实现不完整、内容风格不一致、缺乏验证机制。团队引入 gsd（Get Shit Done）插件，实现了完整的工程化迭代流程。

**gsd 工作流程**：

```
需求启动 → 阶段规划 → 阶段执行 → 循环
```

**各阶段子流程**：

| 阶段 | 子流程 |
|------|--------|
| **需求启动** | 对话完善需求 → 定义范围（v1/v2/out-of-scope）→ 拆分阶段 → 创建路线图 → **（可选）领域调研** |
| **阶段规划** | **（可选）讨论阶段愿景** → 分析阶段目标 → 制定实施方案 → 定义任务清单 → 设置验证标准 |
| **阶段执行** | 执行任务 → 生成产物 → 总结偏差 → 验证确认 |

> **可选步骤说明**：领域调研仅在复杂/陌生领域时需要；讨论阶段愿景仅在需要明确阶段方向时需要。

每个阶段执行后自动进入下一阶段，形成完整闭环，单个需求完成周期从不确定缩短至 **0.5-1 天**。

**质量保障机制**：

| 机制 | 作用 |
|------|------|
| 规范检查器 | 检查 Token 效率、完整性、规范、Bulletproofing、修改影响、结构嵌套 |
| 规范更新器 | 规范与内容冲突时，人工判断是否需要更新规范本身 |
| 双向校验 | 不仅检查变更是否符合规范，也审视规范本身是否需要调整 |

## 3. 经验总结

### 3.1 指令密度悖论

字数与效果不成正比：15000 字指南效果不如 3000 字指南。过度指导导致僵化，Claude Code 将详细步骤理解为"必须执行"而非"参考"。重点淹没于细节，大量信息导致核心目标不突出。

**建议**：构建指南应聚焦核心目标，将详细信息放入 references 按需加载。

**避免这些反模式**：
- ❌ 在 Skill 中写对话模板（如"先问用户 A，再问用户 B"）
- ❌ 为每种异常情况写详细处理规则
- ❌ 在 description 中总结工作流程
- ❌ 把所有信息塞进 SKILL.md（应按需加载）

### 3.2 "做什么" vs "怎么做"

| 对比 | 说明 |
|------|------|
| "做什么" | 定义目标、完成标志、核心原则 |
| "怎么做" | 规定具体步骤、对话模板、异常处理 |

目标清晰 → Claude Code 可自主选择最优路径；步骤详细 → Claude Code 被困在预设流程中。

**建议**：Skill 文档应描述"完成什么"，让 Claude Code 根据上下文判断"如何完成"。

### 3.3 工程化方法

Skill 开发需要软件工程方法支撑：需求分析（明确功能边界）、方案设计（PLAN 文档定义任务分解）、实施验证（SUMMARY 文档记录偏差）、质量保障（规范检查器确保一致性）。

**建议**：将 Skill 开发视为软件开发项目，建立规范和验证机制。

### 3.4 工具选择策略：复杂度驱动

gsd 不是万能良药，需根据问题复杂度选择工具：

| 问题复杂度 | 判断标准 | 推荐方式 |
|------------|----------|----------|
| 高 | 多文件联动、前后逻辑一致性要求、影响现有功能 | gsd 完整流程 |
| 低 | 单文件修改、无上下游依赖、措辞调整 | 直接对话 |
| 中 | 单个流程新增、步骤逻辑调整 | 灵活选择 |

后来团队意识到，gsd 流程同样适用于初版开发。

## 5. 结语

评测 Skill 研发经历了三个关键阶段，核心经验可归纳为：

- **指令设计**：描述目标而非步骤，赋予 Claude Code 自主判断空间
- **工程方法**：建立需求分析、方案设计、实施验证的完整闭环
- **规范演进**：规范与内容双向校验，持续迭代优化
- **工具选择**：复杂问题用 gsd，简单问题直接对话，避免一刀切
