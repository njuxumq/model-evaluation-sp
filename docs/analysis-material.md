# 评测 Skill 研发提效实践

## 1. 研发背景与挑战

### 1.1 评测 Skill 的定位与目标

星评测 2.0 平台为用户提供 AI 模型评测服务，用户可通过前端页面选择评测集、评委模型、评测维度配置，最终提交评测任务并查看结果。然而，这一传统操作模式存在明显的交互门槛：

- **专业概念认知门槛**：用户需理解"评测集"、"评委模型"、"评测维度模板"等专业知识才能完成流程
- **操作路径复杂**：前端页面涉及多个配置步骤，用户需在多个页面间切换
- **错误处理被动**：配置错误时，用户需自行排查，缺乏即时引导

评测 Skill 的开发目标是将上述前端操作流程迁移至 Claude Code 终端对话环境，通过自然语言交互引导用户完成评测任务构建，屏蔽专业知识门槛，实现"几轮对话即可提交评测任务"的体验。

### 1.2 核心职责

评测 Skill 承担两项核心职责：

| 职责 | 说明 |
|------|------|
| 对话引导 | 通过与用户交流，引导用户提供评测场景、评测方式等信息，帮助用户构建评测任务 |
| 任务执行 | 调用后端服务接口，提交评测任务、监控执行状态、获取并向用户展示结果 |

### 1.3 技术挑战

Skill 开发面临的核心挑战：

- **多服务集成**：Skill 需对接认证服务、评测集管理服务、任务管理服务等多个后端接口
- **流程状态管理**：评测任务涉及初始化、构建配置、评测集处理、执行四个阶段，需正确管理阶段状态流转
- **异常处理复杂**：Token 失效、评测集格式错误、任务提交失败等异常场景需妥善处理
- **交互体验优化**：需平衡引导性与灵活性，避免机械式问答

## 2. 踩坑之旅

### 2.1 第一阶段：指令过载的困境（15000 字指南的反噬）

#### 问题背景

开发初期，团队面临"如何开发 Skill"的困惑。虽然有 skill-creator 技能可协助开发，但直接使用存在明显障碍：

- 评测 Skill 涉及多个后端服务、数十个接口，完整信息难以在交互中有效传递
- skill-creator 缺乏评测场景的领域知识，难以设计出符合业务逻辑的流程

为解决这一问题，团队编写了一份详尽的"评测 Skill 构建指南"，期望为 Claude Code 提供充分的背景知识。

#### 构建指南内容

第一版构建指南约 **15000 字**，涵盖：

| 内容类型 | 说明 |
|----------|------|
| 后端服务接口说明 | 所有接口的请求/响应格式、错误码定义 |
| 评测流程设计 | 完整的四阶段流程、任务步骤分解 |
| 异常处理规则 | Token 失效、格式错误、网络异常的处理方式 |
| 脚本设计 | Python 脚本的输入/输出规范、参数定义 |

#### 实际效果

将详尽指南提供给 Claude Code 后，生成的 Skill 存在明显问题：

**问题表现：流程僵硬**

| 现象 | 说明 |
|------|------|
| 按部就班执行 | 用户已提供部分信息，Skill 仍按预设流程逐一询问 |
| 无法灵活跳转 | 缺乏根据对话上下文判断用户意图的能力 |
| 异常处理僵化 | 遄到非预期情况时，无法有效引导用户 |

**根本原因分析**

Skill 文档描述的是"**怎么做**"而非"**做什么**"：

- 过度规定步骤细节，导致 Claude Code 被困在预设流程中
- 详尽的异常处理规则反而限制了灵活性
- Claude Code 将指南理解为"必须严格执行的步骤"，而非"目标导向的参考"

**核心认知转变**

> Skill 应描述"做什么"、"目标是什么"，至于"怎么做"，让 Claude Code 根据对话上下文自主判断。

### 2.2 第二阶段：精简指令的突破（3000 字指南的成效）

#### 调整思路

基于第一阶段的反思，团队对构建指南进行了大幅精简：

**精简原则**

| 原则 | 说明 |
|------|------|
| 聚焦流程目标 | 明确各阶段"需要完成什么"，而非"按什么步骤执行" |
| 放手实现细节 | 不规定具体的对话交互方式、异常处理逻辑 |
| 减少假设性内容 | 只描述已验证必需的信息，不预设可能需要的规则 |

**字数对比**

| 版本 | 字数 | 调整幅度 |
|------|------|----------|
| 第一版 | ~15000 字 | - |
| 第二版 | ~3000 字 | 减少 80% |

#### 第二版构建指南内容

精简后的指南聚焦于：

| 内容 | 说明 |
|------|------|
| 四阶段流程框架 | 初始化 → 构建配置 → 评测集处理 → 执行评测 |
| 阶段完成标志 | 各阶段结束的验证条件（文件存在、接口响应正确） |
| 核心变量定义 | `{work-dir}`、`{session-id}` 等关键变量 |
| 脚本功能说明 | 各脚本的功能定位，不包含参数细节 |

**移除内容**

- 详细步骤分解
- 异常处理规则
- 对话交互模板
- 完整接口定义（移至 references 按需加载）

#### 实际效果

第二版构建指南生成的 Skill 表现显著提升：

| 维度 | 第一版 | 第二版 |
|------|--------|--------|
| 交互流畅度 | 机械式按步骤询问 | 根据上下文灵活引导 |
| 流程跳转能力 | 无法跳过已完成步骤 | 可根据用户意图跳转 |
| 异常处理 | 僵化执行预设规则 | 根据实际情况灵活应对 |
| Token 消耗 | 高（加载大量冗余信息） | 低（按需加载参考信息） |

**效果分析**

团队推测精简指南效果更好的原因：

1. **重点突出**：Claude Code 能清晰理解核心目标，不被细节淹没
2. **自主空间大**：有充分空间根据对话上下文做出判断
3. **Token 效率高**：核心指令紧凑，参考资料按需加载

### 2.3 第三阶段：规范化迭代（gsd 驱动的工程化实践）

#### 问题背景

第二阶段生成的 Skill 已具备基本可用性，但后续迭代开发面临新挑战：

| 问题 | 说明 |
|------|------|
| 需求实现不完整 | 新增功能缺乏完整的前后逻辑关联 |
| 内容风格不一致 | 不同迭代引入的文档风格差异明显 |
| 缺乏验证机制 | 无法确保修改内容与原始需求一致 |

单纯依赖 Claude Code 进行迭代，缺乏软件工程中的需求分析、方案设计、实施验证闭环。

#### 解决方案：引入 gsd 插件

团队调研发现 gsd（Get Shit Done）项目符合需求，引入后实现了完整的工程化迭代流程。

**gsd 工作流程**

```
需求提出 → 阅读项目 → 调研学习 → 对话完善需求 → 方案大纲 → 阶段拆分 → 实施方案 → 执行修改 → 校验确认
```

**完整案例：v1.6 评测方式分支支持**

| Phase | 目标 | 关键产出 |
|-------|------|----------|
| Phase 14 | 评测方式确认 | 任务2结构调整、流程4/5 定义 |
| Phase 15 | 评测点生成流程 | 流程6 定义、四种生成场景 |
| Phase 16 | 文档更新 | 接口定义同步、字段定义补充 |

每个 Phase 的执行包含完整的 PLAN → 执行 → SUMMARY → 验证闭环：

**PLAN 结构**

```yaml
phase: 14-evaluation-mode-confirmation
plan: 01
type: execute
requirements:
  - MODE-01
  - MODE-02

must_haves:
  truths:
    - "用户在任务2步骤2被询问选择评测方式"
    - "选择通用维度级评测后进入流程5"

tasks:
  - 任务1：更新任务2步骤结构
  - 任务2：新增流程4和流程5定义
  - 任务3：更新快速参考表

verification:
  - 任务2结构为三步，步骤2包含评测方式选择
  - 流程4和流程5定义完整

success_criteria:
  - MODE-01：任务2新增评测方式确认步骤 ✓
```

#### 效率提升

| 指标 | 使用 gsd 后 |
|------|-------------|
| 单个需求完成周期 | **0.5-1 天** |
| 需求完整性 | 有明确验证标准 |
| 风格一致性 | 通过规范检查保障 |

#### 适用场景分析

**gsd 的适用场景**

| 场景 | gsd 适用性 | 说明 |
|------|------------|------|
| 复杂功能开发 | ✅ 高度适用 | 需求分析、方案设计、验证闭环可确保质量 |
| 初版 Skill 生成 | ✅ 适用 | 后来团队意识到，gsd 流程同样适用于初版开发 |
| 小需求调整 | ❌ 不适用 | 大材小用，完整流程耗时过多 |
| 紧急修复 | ❌ 不适用 | 流程冗长，不适合快速响应 |

**gsd 不是万能良药**

团队在实践中发现，gsd 的完整流程（需求分析 → 方案设计 → 阶段拆分 → 实施 → 验证）对于复杂问题效果显著，但对于简单问题存在明显的效率损耗：

| 问题类型 | 典型示例 | 推荐方式 |
|----------|----------|----------|
| 复杂问题 | 新增评测方式分支、重构核心流程 | gsd 完整流程 |
| 简单问题 | 修改文档措辞、调整字段名称 | 直接与 Claude Code 对话 |
| 中等问题 | 新增单个流程、调整步骤逻辑 | 根据复杂度灵活选择 |

**关键判断标准**

- 是否涉及多个文件的联动修改 → 使用 gsd
- 是否需要保持前后逻辑一致性 → 使用 gsd
- 是否可能影响现有功能 → 使用 gsd
- 单文件、单点修改、无上下游依赖 → 直接对话

#### 质量保障机制：规范检查器 + 规范更新器

**规范检查器检查项**

| 检查维度 | 检查项 |
|----------|--------|
| Token 效率校验 | 无冗余字段、变量来源明确、目标字数达标 |
| 完整性校验 | 流程步骤完整、用户确认点标注 |
| 规范校验 | YAML frontmatter、语言风格 |
| Bulletproofing 校验 | 禁止行为列表、Red Flags 章节 |
| 修改影响校验 | 功能一致性、向后兼容 |
| 结构校验 | 流程与步骤嵌套是否过深 |

**规范更新器**

| 触发条件 | 处理方式 |
|----------|----------|
| 规范与内容冲突 | 人工判断是否需要更新规范本身 |
| 新模式识别 | 用户确认后纳入规范 |

**双向校验机制**

规范检查器不仅检查变更是否符合规范，也审视规范本身是否需要调整：

```
新增内容 → 规范检查 → 发现冲突 → 人工判断 → 调整内容或调整规范
```

## 3. 经验总结

### 3.1 指令密度悖论

| 观察 | 说明 |
|------|------|
| 字数与效果不成正比 | 15000 字指南效果不如 3000 字指南 |
| 过度指导导致僵化 | Claude Code 将详细步骤理解为"必须执行"而非"参考" |
| 重点淹没于细节 | 大量信息导致核心目标不突出 |

**建议**：构建指南应聚焦核心目标，将详细信息放入 references 按需加载。

### 3.2 "做什么" vs "怎么做"

| 对比 | 说明 |
|------|------|
| "做什么" | 定义目标、完成标志、核心原则 |
| "怎么做" | 规定具体步骤、对话模板、异常处理 |

**Claude Code 需要目标而非步骤**：

- 目标清晰 → Claude Code 可自主选择最优路径
- 步骤详细 → Claude Code 被困在预设流程中

**建议**：Skill 文档应描述"完成什么"，让 Claude Code 根据上下文判断"如何完成"。

### 3.3 渐进式迭代

| 阶段 | 重点 | 说明 |
|------|------|------|
| 第一阶段 | 可用性 | 实现基本功能，验证可行性 |
| 第二阶段 | 流畅性 | 优化交互体验，提升灵活性 |
| 第三阶段 | 完备性 | 工程化迭代，质量保障 |

**建议**：不要追求一次到位，从最小可行版本开始持续迭代。

### 3.4 工程化思维

Skill 开发需要软件工程方法支撑：

| 工程实践 | Skill 开发对应 |
|----------|----------------|
| 需求分析 | 明确 Skill 功能边界和交互目标 |
| 方案设计 | PLAN 文档定义任务分解和验证标准 |
| 实施验证 | SUMMARY 文档记录完成内容和偏差 |
| 质量保障 | 规范检查器确保风格一致和逻辑完整 |

**建议**：将 Skill 开发视为软件开发项目，建立规范和验证机制。

### 3.5 工具选择策略：复杂度驱动

**gsd 不是万能良药，需根据问题复杂度选择工具**

| 问题复杂度 | 判断标准 | 推荐方式 |
|------------|----------|----------|
| 高 | 多文件联动、前后逻辑一致性要求、影响现有功能 | gsd 完整流程 |
| 低 | 单文件修改、无上下游依赖、措辞调整 | 直接与 Claude Code 对话 |
| 中 | 单个流程新增、步骤逻辑调整 | 根据具体情况灵活选择 |

**效率权衡**

| 方式 | 优势 | 劣势 |
|------|------|------|
| gsd 完整流程 | 需求完整、逻辑一致、有验证保障 | 流程冗长、耗时较多 |
| 直接对话 | 快速响应、灵活调整 | 缺乏验证、可能遗漏 |

**建议**：建立问题复杂度判断意识，避免"一刀切"地使用或排斥某种工具。

## 5. 关键实践建议

### 5.1 构建 Skill 的核心原则

1. **Description = 触发条件**：绝不总结工作流程
2. **Token 效率优先**：按需加载，引用不重复
3. **验证驱动**：No document without failing test first

### 5.2 文档字数目标

| 文档类型 | 目标 | 原因 |
|----------|------|------|
| SKILL.md | < 5000 字 | 高频加载 |
| 阶段文档 | < 2000 字 | 阶段触发时加载 |
| 流程文档 | < 1500 字 | 按需加载 |

### 5.3 Progressive Disclosure 设计

| 级别 | 内容 | 加载时机 |
|------|------|----------|
| 元数据 | name + description | 始终在 context |
| SKILL.md | 核心指令和工作流 | Skill 触发时 |
| references/ | 详细信息 | Claude 判断需要时 |

## 6. 小结

评测 Skill 的研发过程经历了三个关键阶段：

1. **第一阶段**认识到过度指导的弊端，理解了"做什么"与"怎么做"的本质区别
2. **第二阶段**通过精简指令实现了交互流畅性突破
3. **第三阶段**引入 gsd 实现工程化迭代，建立了规范与验证双重保障机制，并意识到 gsd 同样适用于初版开发

核心经验可归纳为：

> **指令设计**：Skill 文档应描述目标而非步骤，赋予 Claude Code 自主判断空间；
> **工程方法**：Skill 开发应遵循软件工程方法，建立需求分析、方案设计、实施验证的完整闭环；
> **规范演进**：规范与内容应双向校验，持续迭代优化；
> **工具选择**：根据问题复杂度选择开发方式，复杂问题用 gsd，简单问题直接对话，避免一刀切。
